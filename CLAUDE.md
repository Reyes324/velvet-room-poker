# 翡翠厅 — 项目规则

## SDD 优先工作流

SDD 文件位于 `openspec/changes/online-texas-holdem/`，包含 `design.md`（决策与场景）和 `tasks.md`（任务清单）。

**每次有设计决策、功能变更、功能取消时，必须按以下顺序执行：**

1. 先更新 `design.md` 和/或 `tasks.md`
2. 再写代码
3. commit 包含 SDD 更新

不等用户提醒，主动执行。

### 什么时候更新 SDD

| 变更类型 | 更新位置 |
|---|---|
| 取消/移除功能 | tasks.md 标 `~~cancelled~~` + 记原因；design.md 更新场景 |
| 设计决策变更 | design.md Decisions 节 |
| 新需求从对话浮现 | design.md Open Questions → 确认后加 tasks.md → 再写代码 |
| 任务完成 | tasks.md 勾选 `[x]` |
| 微调 UX（颜色/动画） | 不用更新，commit message 记录即可 |

## Git 规则

- 不经用户明确指令，不自动 push
- commit message 用英文，格式：`type: description`

## 工作优先级

- **正确性 > 视觉打磨**：这是一个真的要给朋友联机打牌用的项目，不是 demo。除非用户主动提 UI/视觉需求，否则顺序永远是：正确性 bug → 测试覆盖 → 真实浏览器验证 → 最后才是视觉/动画打磨。
- **改动要结构化，不要打补丁**：收到改动需求（尤其是一批反馈）时，先找共性根因/所属的状态机或概念，再动手改，不要见一个症状加一个 if/一处判断。改完之后如果发现某处是"哪里疼医哪里"式的补丁（比如冗余判断、同一个常量在多处各存一份、状态被从类外部直接塞进对象属性而不是通过类自己的方法管理），要顺手理清楚，不要留着。
- **判断"两个改动是否会互相影响"这类问题时，不接受纯手算/读代码就下结论**——要么跑真实端到端测试，要么用 Playwright 之类工具实测（比如量真实渲染出来的 bounding box），再下结论。这条项目里已经因为"手算 CSS 觉得没问题，实测发现真的重叠"踩过坑。
- 判断一项任务是否"做完了"，不能只看 `tasks.md` 的勾选状态——要读代码、跑测试、真实跑一遍浏览器确认，checklist 可能没跟上实际进度。
