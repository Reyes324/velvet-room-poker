## Context

全新项目，从零构建一个支持多人在线实时对战的德州扑克 Web 应用。目标是轻量、无需注册，通过分享链接即可开始游戏。玩家数量 2-9 人，需要低延迟的实时状态同步。

## Goals / Non-Goals

**Goals:**
- 浏览器直接访问，无需安装任何客户端
- 房主创建房间并分享链接，朋友点击即可加入
- 完整德州扑克规则（No-Limit Hold'em）
- 实时同步所有玩家操作和牌局状态
- 移动端友好的响应式 UI

**Non-Goals:**
- 用户账号/登录/持久化排行榜
- 真实货币/积分系统
- AI 对手（Bot）
- 观战模式
- 游戏进行中的断线重连（MVP 阶段，断线自动 fold，下局重连）——注意：这条只覆盖"游戏进行中"；大厅/等待阶段的断线有独立的宽限期处理，见下方「大厅断线宽限期」决策，两者不是一回事
- ~~游戏中途加入新玩家（游戏进行中拒绝加入，等下局）~~ —— 已被下方「游戏中途加入 + 筹码归零个人决策」决策取代：用户实测反馈希望房主开局后朋友仍能通过链接加入，改为允许中途加入、从下一手自动入座
- ~~观战模式~~ —— 中途加入等待入座、筹码归零后选择"旁观留下"，两者都需要"能看牌局但不参与"的旁观视图，因此这条 Non-Goal 被下方决策顺带实现，不再是有意排除的范围；仍然不做的是"纯观战、从不下场"的独立入口（没有人会以旁观者身份创建/加入房间——旁观状态只能从"中途加入等待"或"筹码归零"这两个具体场景派生）
- 行动超时倒计时对其他玩家可见（延后决策）

## Decisions

### 前后端架构：单体 Node.js + React

**选择**: Express + Socket.io 后端，React 前端，单仓库部署。

**理由**: Socket.io 提供开箱即用的 WebSocket + 降级支持，适合小团队快速开发。单体部署简化运维，无需分离前后端服务。React 便于管理复杂的实时 UI 状态。

**备选方案**: Next.js（SSR 复杂度不必要）、纯 WebSocket（需自行处理重连/心跳）。

---

### 游戏状态管理：服务端权威

**选择**: 所有游戏逻辑和状态只在服务端计算，客户端只渲染和发送操作指令。

**理由**: 防止作弊（客户端不能看到其他玩家手牌）、避免状态不一致。服务端是唯一真相来源。

**备选方案**: 客户端计算 + 服务端验证（逻辑重复，风险更高）。

---

### 房间标识：6 位纯数字码

**选择**: 6 位随机数字（如 `482931`）作为房间 ID。

**理由**: 纯数字比字母数字更容易口头分享（"四八二九三一"），朋友局场景下常见。原方案是字母数字混合，经用户反馈后改为纯数字。

---

### 数据结构：游戏状态对象

服务端维护每个房间一个 `GameState` 对象：

```
GameState {
  roomId, phase (waiting/preflop/flop/turn/river/showdown),
  players: [{ id, name, chips, holeCards, bet, status, isDealer }],
  communityCards[], pot, currentBet, activePlayerIndex,
  deck (服务端私有，不广播)
}
```

广播给玩家时，隐藏其他玩家手牌（只发送 `{ hidden: true }`），只对摊牌阶段或弃牌以外的玩家展示。

### S1：正常开局
1. 房主打开链接 → 输入昵称 → 创建房间 → 复制邀请链接发朋友

### S3：筹码归零 —— 已被下方「筹码归零 Rebuy（借一底）」章节取代并实现，本节仅保留历史背景
- 玩家输光筹码 → 当局结束后自动移出下一局
- 触发 `game:ended` 事件 → 客户端显示结束消息
- ~~未来可扩展：重新买入（rebuy）按钮，暂不实现~~ **已实现**，见「筹码归零 Rebuy（借一底）」章节
- **Bug 修复记录**：`nextRound()` 因筹码不足结束时，服务端曾经只广播 `game:ended`、没有再广播 `room:state`，导致大厅里落败玩家看不到自己最新的 0 筹码和"+借一底"按钮（要手动刷新才会更新）。已修复为 nextRound 后统一广播 room:state，见 tasks.md 10.2

- 新玩家尝试加入进行中的房间 → 返回错误"游戏已开始，无法加入"

### S6：行动超时 ~~（已取消）~~
- **决策：整个超时机制已移除** — 用户明确不需要，朋友局场景下强制限时体验差

### 视觉信息层级与配色（v3 预览评审）

**问题**：早期每个元素（位置角标、下注、后手、底池、街道）都是"带边框的小药丸"，且颜色多（角标白/黄/蓝），无主次，视觉嘈杂。

**决定（建立单一层级，去边框）**：
- **底池**：去掉边框/背景，纯发光金色大数字浮在桌面，作为唯一视觉焦点。
- **街道指示、后手**：去边框，纯文字。
- **下注筹码**：唯一保留"筹码感"的元素——一个金色圆形筹码 token + 数字，无外框。
- **位置角标 D/SB/BB**：统一为单一象牙色"按钮"样式（取消白/黄/蓝三色），小尺寸贴头像边缘。
- **金 vs 冷色分工**：金色专用于"在玩的钱"（底池、下注）；**后手（stack）改用冷色 platinum `#9FB0AC`**，作为辅助信息退后一层，与金色拉开差距。
- **金额字体两级层级**：
  - **主角数字（底池、结算赢得金额）用展示衬线体 `DM Serif Display`** —— 敦实优雅，作为全场焦点数字有高级牌局质感。
  - **功能性小额（后手、下注、余额、结算明细）保持等宽体 `Space Mono`** —— 对齐工整、读作"账本/钱"。
  - 这是有意的两级层级（焦点数字优雅 / 功能数字工整），非"随便混字体"。演进过程：Cinzel（数字花哨被否）→ 一度全 Space Mono（底池放大偏技术感）→ 现选 DM Serif Display 给主角数字。
  - Cinzel 仅保留给装饰性标题与房间码（非金额）。

### 扑克牌牌面设计（v3 预览评审）

**选择**：牌面升级为接近真实扑克牌的高级质感。

**关键细节**：
- **对角双角标**：左上 + 右下（旋转 180°）各一组"点数 + 小花色"，符合真牌惯例（原仅左上单角标显得简陋）。
- **无边框**：曾加内描线框，用户不喜欢框，已去掉——牌面只留暖白纸面 + 角标 + 中心花色，干净（即样式 A）。
- **暖白纸面**：象牙白渐变 + 内高光，配合更讲究的投影。
- **花色配色**：红 `#B3271E`（更沉稳）、黑 `#1F2A33`（略冷的近黑）。
- **点数字体用 `Oswald`（窄体）**：解决 Georgia 下两位数"10"偏扁的问题，condensed 让数字紧凑利落；花色符号仍用原字体渲染。
- 牌背：深绿渐变 + 金色 ✦ 暗纹。
- **弃牌玩家**：不再整体半透明（会透出桌面，观感差），改为**实心、去饱和、压暗**的头像 + 下方"弃牌"文字说明。视觉清晰，用户明确认可。
- 翻牌前不显示公共牌占位实底（牌未发出），改为极淡的虚线空位，减少"框框"噪声。

### 样式架构：设计系统为唯一源，预览即状态画廊（方案 B）

**问题**：早期 `states-preview.html` 预览**脱离 design token 体系**单独手调，调出的最终值困在预览内联 CSS 里，没沉淀回 tokens/组件；生产组件又用另一套 class（`card--lg` 等）"翻译"预览，每次翻译走样，预览失去"一次确认直接落地"的意义。

**选择（方案 B）**：
- **唯一事实源 = `tokens.css`（值） + 共享组件 CSS**（`client/src/styles/velvet.css`，用预览那套 class，引用 token）。
- **预览 = 状态画廊**：`states-preview.html` 改为 link 共享 CSS、用同样 class 摆出所有状态，不再内联另写一份。
- **生产 = 同一套 CSS + 数据**：Card/PlayerSeat/RoomPage 直接 import 共享 CSS、输出预览的 class 与 DOM 结构。
- 三者同源，改一处 CSS/token，画廊与生产同步，永不翻译。

**落地步骤（一次性返工）**：① 把已确认预览的最终值回收进 tokens + `velvet.css`；② 预览改造为画廊；③ 生产组件改用预览 class + import velvet.css。

**理由**：用户的诉求是"轻量预览确认设计 → 直接复制进生产"。只有让生产**消费**预览的 CSS（而非重写），该工作流才成立。详见协作手册「设计迭代要在设计系统上做」。

**备选（未选）**：方案 A —— 直接把预览 `<style>` 当生产 CSS，token 退役。更快但丢失 token 抽象（语义色名/主题化），用户看重 token 故选 B。

### 筹码归零 Rebuy（借一底）

**决策（Q1 已确定）**：采用**选项 B** —— 支持 rebuy，玩家输光后留在房间，在大厅等待阶段自助借一底。

**规则**：
- 一底 = ¥1,000（与初始买入相同）
- 每次 rebuy += 1,000 chips，debt += 1,000（无上限，由玩家自行决定）
- 时机：~~仅在 waiting 状态（大厅）下允许 rebuy，游戏进行中不可借~~ —— 已被下方「游戏中途加入 + 筹码归零个人决策」决策取代：门禁改成"本人 chips===0 即可借，跟房间整体状态无关"，因为 3 人以上桌子里游戏进行中归零的玩家在旧规则下要等到整桌结束才能借，体验缺口太大
- 仅输光筹码（chips=0）的玩家才会看到 rebuy 按钮（本人可见）
- debt > 0 时，所有人都能在大厅 + 游戏桌看到该玩家的"借了¥X"债务状态

**数据模型**：在 `players` 对象添加 `debt: 0` 字段，通过 `getLobbyState()` 广播给所有玩家。

**nextRound 行为变更**：不再自动移除 chips=0 玩家，改为仅把有筹码的玩家传入 `GameEngine`；chips=0 玩家在大厅旁观等待 rebuy，rebuy 后下一局参战。

**UI**：
- 大厅：chips=0 时本人看到"+借一底"按钮；全员看到"借¥X底"橙色 badge
- 游戏桌：PlayerSeat 暂不展示 debt（大厅展示已足够；避免游戏桌信息过载）

### 移动端设计规范：Scale-to-Fit（统一适配方案）—— ~~已被下方「顶部/底部贴边 + 仅中间牌桌缩放」取代~~，本节仅保留历史背景

**选择**：以 375×812（iPhone X）为标准设计画布，通过 JS 计算 scale 系数（`Math.min(vw/375, vh/812)`）自适应所有设备，`vh` 取当前实际可见视口高度（`visualViewport.height`，回退 `innerHeight`）。

**规则**：
- 游戏画布固定 375×812，所有绝对坐标相对此画布
- `vh/812`（完整高度，不裁切）：scale 必须让整个 375×812 画布完全落在当前可见视口内
- `vh` 必须用「当前实际可见」的高度，不能用 `window.screen.height`（物理屏幕高度）——地址栏/浏览器 UI 出现时 `screen.height` 仍是满屏值，比真正可见区域（`innerHeight`/`visualViewport.height`）大，会算出偏大的 scale，把画布顶部推出可见视口之外（`.stage-wrap` 是底部锚定，超出部分从顶部溢出）。单挑（1v1）时唯一的对手座位正好在椭圆桌面正上方顶点，是第一个消失在屏幕外的元素
- `transform-origin: bottom center` + `.stage-wrap { align-items: flex-end }`：游戏从底部锚定，确保操作栏（action bar）在任何设备上始终可见
- 同时监听 `resize`、`orientationchange`、`visualViewport.resize`，地址栏出现/隐藏时重新计算 scale
- 适配结果：iPhone SE（375×667）有约 17px 侧边空白（可接受），现代机型（390×844+）填满屏幕；地址栏可见时画布整体等比缩小而非被裁切

**背景**：早期版本存在两个根本性布局问题：
1. `RoomPage.css` 中的旧 `.lobby { min-height:100dvh }` 与 velvet.css 的绝对定位大厅打架，导致"开始游戏"按钮被推出屏幕
2. 移动端没有统一的适配规范，不同屏幕表现不一致

**移除 RoomPage.css**：发现 `RoomPage.css` 是旧设计系统遗留的冲突文件（`.lobby-code`、`.start-btn`、`.table-view` 等已废弃 class），全部移除，Toast 样式迁入 velvet.css，统一单源架构。

**踩坑记录**：中途曾改用 `window.screen.height` 替代 `innerHeight`（意图是让地址栏出现/隐藏时 scale 不跳动、避免小屏出现黑边），并同时把裁切余量从 `vh/712` 收紧为 `vh/812`。这两个改动叠加后，只要浏览器地址栏可见（`screen.height > innerHeight` 的场景，移动端打开页面的常态），scale 就会算大，画布顶部（含单挑唯一对手座位）整段推出可见视口——即用户反馈的"单挑时对手内容飘出手机屏幕"。修复：`vh` 改回读实际可见视口（`visualViewport.height` 优先，`innerHeight` 兜底），让画布等比缩小以完整容纳，而不是用偏大的物理尺寸硬算再被裁切。

### 移动端布局重构：顶部/底部贴边 + 仅中间牌桌区域缩放（用户实测反馈第八轮，2026-07-19）

**背景**：上面的 Scale-to-Fit 方案有个没意识到的副作用——用户反馈手机上牌桌两侧有明显留白，宽度没拉满。原因：`.game-stage` 是整个 375×812 画布作为一个整体用单一 `scale = Math.min(vw/375, vh/812)` 缩放。如果浏览器（尤其是某些 App 内置浏览器）上下 UI 常驻占用空间较多，可视区域的宽高比会比 375:812 更"矮胖"，`vh/812` 会小于 `vw/375`，缩放取小值时高度是瓶颈，宽度被迫跟着一起缩小——两侧因此留白，即使宽度本来有富余空间没用上。

**选择**：不再对整个画布做单一等比缩放，改成"顶部状态栏、底部操作栏各自按自己的实际内容渲染、贴边显示，宽度永远等于视口宽度；只有中间的牌桌区域（椭圆桌+座位+英雄手牌）跟随剩余空间弹性缩放"。本质是从"整体 transform:scale"改成"顶固定高度 + 底固定高度 + 中间 flex:1 弹性区域"的标准 flex 布局。

**实现**：
1. `.game-stage` 从固定 375×812 尺寸 + `transform:scale()` 改成 `width:100%; height:100dvh; display:flex; flex-direction:column;`
2. `.top-bar` 从绝对定位改成真实 flex 子元素（`flex:0 0 auto`），`padding-top` 用 `max(env(safe-area-inset-top,0px),14px)` 处理刘海屏安全区，永远贴顶、永远满宽
3. 新增 `.table-zone`（`flex:1 1 auto; min-height:0; overflow:hidden;`）+ `.table-canvas`（固定 375×`TABLE_REF_H`(610) 的设计画布，通过内联 `transform:scale()` 缩放并居中）——牌桌椭圆、座位、英雄手牌全部挪进这个新画布，`GameTable.jsx` 新增 `useTableScale(ref, refW, refH)` hook，用 `ResizeObserver` 直接测量 `.table-zone` 的实际渲染尺寸算缩放系数，不再依赖 `window.innerWidth/innerHeight` 或 `visualViewport`——这一层的旧坑（`screen.height` vs `innerHeight`）从根上不存在了，因为缩放对象从"整个视口"变成了"一个真实测量到的 DOM 容器"
4. `.action-bar`/`.waiting-bar` 同理改成 flex 子元素（`flex:0 0 auto`），永远贴底、永远满宽，`padding-bottom` 叠加 `env(safe-area-inset-bottom)`
5. `.lobby`（大厅玩家列表）同样从绝对定位改成 `flex:1 1 auto`——本来就是纯文字列表，不含需要保持比例的几何绘制，不需要参与缩放，直接享受原生响应式布局
6. `seatPositions()` 的椭圆参数（`cy`）、`table-oval` 顶部偏移、`hero-section` 的 `bottom` 值、`CARDS_SIDE_BELOW_Y` 阈值，都因为参考画布从 812 缩到 610（顶部状态栏不再占用画布内空间）做了等比重新推导

**意外的架构收益**：`.action-bar` 现在是牌桌画布**外面**的独立 flex 兄弟节点，展开加注面板导致操作栏变高时，会自动挤压 `.table-zone` 的可用高度、让整个牌桌画布等比缩小，物理上不可能再出现"加注面板盖住英雄手牌信息条"——第二/三轮反馈里为了避免这个重叠专门做的像素级精调（`hero-section: bottom:178px→148px` 来回调整、压缩加注面板高度）现在不再需要手动兜底，是这次重构顺带解决的。

**验证**：Playwright 4 种视口（含完全复现用户截图问题的"浏览器 UI 占用高度"430×777 场景，以及更极端的 430×650）下确认顶部/底部宽度精确等于视口宽度、零元素溢出可见区域；加注面板展开截图确认全宽显示且不再与手牌重叠。

### 顶部对手头像仍会飘出 + 牌桌区域改成横纵独立缩放（用户实测反馈第九轮，2026-07-19）

**背景**：上面的重构上线后用户反馈"中间顶部的头像又有点飘出去"，同时提出一个更根本的问题：牌桌区域自己（`.table-canvas`）内部仍然用的是单一 `Math.min(w/refW, h/refH)` 等比缩放——跟这一整轮问题的根因（顶部/底部贴边重构之前，整页缩放同样的算法）完全一样，只是把"两侧留白"这个症状从整个页面收缩到了牌桌这一个区域内部。当浏览器 UI 占用高度较多、高度是瓶颈时，牌桌区域两侧一样会重新出现留白。

**头像飘出根因（用 Playwright 量出来的真实数字，不是猜的）**：`seatPositions()` 的椭圆 `cy=215, ry=195`，最顶上那个对手座位中心点落在 `cy-ry=20`，跟 `table-oval` 的 CSS 顶部偏移 `top:20px` 精确相等——而这个 20px 同时也几乎正好等于头像本身的半径（默认头像 40px 直径，半径 20px）。也就是说，最顶上那个头像的设计半径和到画布顶边的留白**从一开始就是零边距**，任何一点亚像素取整、边框宽度、位置徽章，都会把它推到 `.table-zone` 的 `overflow:hidden` 裁切线以上。实测在高度是瓶颈的视口下（如 430×777、430×650、375×667）该座位真的会被裁掉 ~1px——量级虽小，但证明这不是设备特例，是画布尺寸设计上就没留够余量，在越极端的视口下只会越明显。

**选择**：
1. 牌桌区域缩放从"单一等比 `scale(s)`"改成"横向、纵向各自独立缩放 `scale(scaleX, scaleY)`"——`scaleX` 永远等于容器实际宽度 ÷ 参考宽度（牌桌横向永远贴满，不会再有留白），`scaleY` 独立按容器实际高度 ÷ 参考高度计算，不再取两者较小值。代价：在跟设计稿参考比例（375:610）差异较大的极端视口上，头像会有肉眼可辨但不夸张的轻微椭圆化（实测最极端场景约 20% 形变），换来牌桌横向永远填满——用户已在这一轮反馈中明确认可这个取舍（"宽的时候可以扁一点，窄的时候可以瘦一点"）。
2. `seatPositions()` 的 `ry` 从 195 收紧到 180（`table-oval` CSS 的 `top`/`height` 同步从 20/390 改成 35/360），把最顶上那个座位的中心点从 y=20 推到 y=35，给头像半径腾出约 15px 的真实缓冲，不再是设计上的零边距。`rx`（横向半径）、`cy`（中心点）不变，牌桌整体形状只是纵向略微更圆一点，肉眼几乎看不出差异。

**验证**：Playwright 4 种视口 × 双方视角实测头像包围盒坐标（不是目测），修复前在 3/4 视口下最顶上的对手头像确有 ~1px 裁切，修复后全部归零；同时验证牌桌椭圆左右两侧留白始终等于设计稿里 18px 按 `scaleX` 换算后的固定内边距（跟视口宽度成正比、左右对称），而不再是缩放算法带来的、跟高度瓶颈相关的不对称留白。

### 撤销「牌桌内容随非等比缩放变形」的取舍 + 菜单按钮放大（第十二轮，2026-07-19）

**背景**：上一条决策里用户认可的取舍——浏览器往外拉伸、`scaleX`≠`scaleY` 时头像轻微椭圆化——实测下来并不是"轻微"：`.table-canvas` 上的 `scale(scaleX, scaleY)` 作用于整个子树，不只是头像，牌、底池数字、街段标签、筹码文字全都跟着两轴不同比例被拉伸/压扁，浏览器越往宽拉变形越明显。用户明确表示这样不对：牌桌圈（位置/形状）可以拉宽，但牌、数字、标签这些具体内容不能变形。等于推翻上一轮"用户已认可此取舍"的决定。

**选择**：把"位置铺开"和"内容渲染"拆成两层，不再共用同一个非等比变换：
1. 位置层不变——`.table-canvas` 依旧用 `scale(scaleX, scaleY)`，座位/牌桌圈的位置和形状继续非等比铺满容器（牌桌横向永远贴满这个既有优点保留）。
2. 内容层新增统一缩放 `scaleUniform = Math.min(scaleX, scaleY)`，并算出反向抵消系数 `csx = scaleUniform/scaleX`、`csy = scaleUniform/scaleY`（写成 CSS 变量 `--csx`/`--csy`）。凡是渲染实际内容的节点（牌桌内容包裹层 `.table-oval-content`、每个座位 `.player-slot`、`.hero-section`）在继承父级非等比缩放的基础上，追加 `scale(csx, csy)`——两次缩放相乘后视觉上等价于单一的 `scale(scaleUniform, scaleUniform)`，牌、头像、文字因此始终保持原比例，同时位置仍然跟随非等比的 `scaleX/scaleY` 铺开。
3. 代价：`scaleUniform` 取两轴较小值，极端宽高比视口下内容会比"贴着某一轴拉满"更小一些（周围留白变多），但换来的是牌桌上任何文字/图形都不再变形——这次明确是用户要的取舍方向，跟上一轮相反。

**顺带修复**：`.menu-btn`（左上角菜单按钮）从 32×32px 放大到 44×44px（触屏最小可点击尺寸的行业标准），字号从 15px 提到 20px。

### 游戏中途加入 + 筹码归零个人决策 + 账本视图（用户实测反馈第十轮，2026-07-19）

**背景**：用户提出两个交互问题。(1) 房主开局后，朋友能否仍通过邀请链接加入，从下一手自动入座？现状是硬拒绝（`Non-Goals` 原先明确排除这条）。(2) 某玩家筹码归零后，现在的行为是"等到全桌只剩 <2 个有筹码的人时，整个房间掉回大厅，游戏对所有人中断"；用户希望改成"只有归零本人看到一个决策弹窗（借一底 / 旁观留下 / 离开），其他人的牌局不受影响地继续（前提是还有 ≥2 人有筹码——单挑归零依然只能整体暂停，这是德州扑克规则本身决定的，不是要修的 bug）"。用户同时要一个账本视图（初始筹码/已借入/当前筹码），随时可查，不限于大厅。

**排查发现（两轮 Explore + 一轮 Plan 子代理逐行核实源码）**：`GameEngine.js` 的 `players` 数组是每手从 `Room.players` 重新过滤构建的一次性快照，从未支持、也不需要支持"插入正在进行的一手"——两个功能的插入点天然都在"两手之间"，因此 `GameEngine.js` 和 `server/index.js` 完全不需要改动，改动全部集中在 `RoomManager.js` 的两处门禁 + 客户端渲染路径。过程中发现一个已经能在生产环境触发的潜伏 bug：`GameTable.jsx` 的 `const me = ordered[0]`（未检查"我是否真的在这一手玩家列表里"）在归零场景下会把某个真实对手的座位错误地渲染成"我"（名字标错、位置徽章全错），而不是报错或掉回大厅——这两个新功能会让这个潜伏 bug 从"几乎撞不到的边界情况"变成"每次归零/中途加入都触发"，本轮一并修复。

**选择**：
1. `Room.addPlayer`（`RoomManager.js`）去掉"仅 `status==='waiting'` 才能加入"的拒绝，只保留 9 人满座 + 重复 id 检查；新玩家照常 `chips:1000`，`nextRound()` 本来就按 `chips>0` 从 `this.players` 全量重建下一手的参战名单，中途加入的人自动从下一手生效，不需要新增"待入座"标记字段。
2. `Room.rebuy`（`RoomManager.js`）门禁从"仅 `status==='waiting'`"改成"仅本人 `chips===0`"，与房间整体状态无关——见上方「筹码归零 Rebuy」决策的更新。
3. 顺带修复庄位轮转的潜伏 bug：`dealerIndex` 是对每次重新过滤出的 `active` 数组取模，数组大小/顺序因中途加入或归零排除而变化时庄位会不连续地跳。改成记 `dealerId`（上一手庄家的玩家 id），下一手在新数组里找这个 id 的位置、庄位顺延到其后的下一个座位；找不到（该玩家已离场）则退回座位 0。这个改动不属于用户原始请求，是本轮排查中顺带发现并修复的，因为这两个新功能会让牌桌人数变动的频率大幅提高，不修的话下一次实机测试大概率会把它当成新 bug 报回来。
4. `Room.getLobbyState()` 新增 `startingChips` 字段，账本视图直接读取，不在客户端重复维护一份 1000 常量。
5. 客户端：`RoomPage.jsx` 新增 `amPlaying`（我是否在当前这一手 `gameState.players` 里）判断，替代过去"有 `gameState` 就当作我在游戏里"的隐式假设；`GameTable.jsx` 新增旁观渲染路径（不再有效座位时不得回退到 `ordered[0]`），座位几何新增一个"没有英雄锚点、把所有人均匀分布在完整椭圆上"的变体；筹码归零的检测用 `prevChipsRef` 追踪本人筹码 `>0→0` 跳变触发决策弹窗，跟本次会话之前给发牌动画加的 `justDealt`/`justRevealed` 判定是同一套手法。选择"旁观留下"后，footer 保留一个常驻的"+借一底"入口（不会被卡死出不来）。
6. 顶部 ≡ 菜单从"点击直接弹退出确认框"改成两行小弹层（账本 / 退出游戏），`GameTable.jsx` 和 `Lobby.jsx` 两处入口保持一致。

**Non-Goals 联动**：见上方 Non-Goals 小节，"游戏中途加入新玩家"和"观战模式"两条已相应划掉。

**验证**：服务端单元测试（75/75）覆盖 addPlayer 中途加入、rebuy 的 chips===0 门禁正反两个方向、dealerId 轮转在"庄家中途归零离场"场景下的正确性。

**踩坑记录：这个沙盒环境的多 page 硬限制**——实现时想按最初计划写 3 个真实玩家身份的 Playwright 场景（已开局 2 人 + 中途加入的第 3 人），结果发现这个沙盒环境存在一个跟本次改动完全无关的限制：单个测试进程里第 3 个浏览器 page（不管是否属于同一个 context）几乎总是卡死在 `page.goto` 上——用 3 个完全不含任何游戏逻辑的空白页复现出同样的结果，确认是环境本身的资源上限（很可能是代理层的并发连接数限制），不是应用代码问题；这也解释了旧版 S4 测试（3 个 context）为什么本来就是本 session 记录在案的已知偶发失败之一。应对方式：① 服务端集成验证改用 socket.io 服务端自带的 `/socket.io/socket.io.js` 客户端脚本，在已有的 page 里开一条裸 socket 连接模拟"第三个玩家"，只用 2 个真实 page 就验证了 `room:join` 在游戏进行中真的会被服务端接受、返回 1000 筹码、不在当前这一手里；② 客户端旁观渲染路径（不会把座位错标成"我"、footer 三态、两个新弹窗）改用项目已有的 `?states=N` 开发自检画廊（`fixtures.js`/`StatesGallery.jsx`），单 page 直接给 `GameTable`/新组件喂真实 props，覆盖力度不打折扣，还避开了这个环境限制；③"归零玩家单独借入、其他人不受影响"这条由 `RoomManager.test.js` 的专门单测覆盖（比端到端 UI 测试更精确，也完全不受这个环境限制影响）。

### 行动指示气泡统一 + 发牌质感/动画补全（第十三轮，2026-07-19）

**背景**：用户反馈三个问题。(1) "轮到谁"的雷达波纹圈视觉上像"一个断续、线框不完整的圈"，不想要，要求统一改成气泡样式，且气泡不应该像现在这样会消失。(2) 牌背太薄，感觉不出"派了牌"。(3) 公共牌/对手手牌的发牌动画不够有"发牌"的动作感，而且对手手牌发完之后就彻底看不见了。

**排查发现**：(1) 读代码发现实际上存在两套独立机制——对手下注一直有一个不会消失的持久 `.bet-chip` 气泡，但 hero 自己下注只有一条 1.6 秒后淡出的文字提示，从来没有对应的持久气泡；"不同人眼中不统一"说的正是这个不对称，不是幻觉。(2) `PlayerSeat.jsx` 里对手的两张背面小牌只在 `dealing`（发牌那一瞬间，约 1 秒）渲染，动画播完 `dealing` 变 false 后这段 JSX 直接不渲染了，直到摊牌才重新出现——"对手好像没有牌"是真实的展示 bug，不是错觉。

**选择**：
1. 去掉 `.seat.is-active::before/::after`"雷达波纹圈"整套（含 `@keyframes activeRing`），头像本身的描边发光（`.is-active .avatar` 的 border+box-shadow 脉冲）保留——那是另一套机制，不是用户描述的"圈"。
2. `GameTable.jsx` 给 hero 补上跟对手完全同源的 `.bet-chip`（同一套"朝向池心方向平移"的计算），从此 hero 自己下注也有一个不会消失的持久气泡，跟对手视角完全一致。文字类的"说话"气泡（跟注/加注/弃牌/过牌/ALL IN）保持原有的淡出行为不变——用户明确说的是"气泡表示放筹码"这一种要改成不消失，不是所有反馈都要常驻。
3. `.c-back` 从纯色渐变+单个星标，加上内嵌高光/暗部阴影模拟出厚度（不是纯平矩形），再叠一层低透明度（4-5%）菱格纹理——是材质模拟，不是页面装饰条纹，且只作用于 28-84px 的卡片元素本身。
4. `cardDeal` 动画从"原地缩放淡入"改成"从右上方带旋转位移滑入再摆平"，更接近发牌手部动作的观感；社区牌本来就有的逐张交错延迟（`communityDealDelayFor`）沿用不变，这次只是让每一张落地时的动作更明显。
5. ~~对手的两张背面小牌改成整手牌局期间常驻显示~~ **已被 2026-07-21 决策取代**：贴边双栏骨架上线后暗牌持续跟下注筹码/公共牌抢中间过道空间，用户反馈这两张牌摊牌前不传递任何信息、纯粹占地方，改成摊牌前完全不渲染，只在真摊牌时显示（见后文「单挑大小盲分配错误 + 摊牌前对手暗牌下线」）。（`hasCards = gamePhase !== 'waiting'`），只在刚发牌那一下播放飞入动画，之后静止显示，弃牌后隐藏，摊牌时切换成揭示样式（原逻辑不变）。
6. 连带问题：手牌常驻之后，会跟同样默认贴在头像正上方的"说话"文字气泡打架。新增 `bubbleStyle()`：手牌在上方时文字气泡堆叠到手牌上方；手牌因为贴近顶部边缘被推到侧边时（`cardsSide` 命中），文字气泡改用头像正上方的默认位置——两者互补，不会同时占用同一个位置。
7. 密集桌（7-9 人）连带问题：Playwright 实测 9 人桌场景后发现相邻座位的常驻手牌会互相重叠甚至被裁到视口外——`.game-stage--dense` 新增头像 38→34px、手牌 28×40→21×30px、`.reveal` 间距收紧的密集桌专属尺寸，重叠消除。

**验证**：`?states=` 自检画廊逐态截图核对（含临时插入又移除的"密集桌测试"9 人场景、临时插入又还原的 hero 下注场景），控制台无报错；受限于自检画廊只渲染单帧静态 state（`actionBubbles` 靠运行时状态变化才会产生，静态 fixture 天然产生不了），气泡与常驻手牌"不重叠"这条主要靠代码走查 + 上面第 6 点的互补定位逻辑保证，未能在自检画廊里截到两者同框的真实帧，后续接入真实房间实测时应留意确认。

### 修复发牌动画从未真正播放过的 bug，并纠正上一轮的错误结论（第十七轮，2026-07-19）

**背景**：用户反馈"公共牌没有动画"。这轮之前（第十六轮，见 tasks.md 24.3）曾用 Web 搜索核实过德州扑克真实发牌规则（从小盲开始、顺时针、一次一张、两轮、公共牌最后发），对照代码后得出"发牌顺序本来就完全符合规则，不用改"的结论——**这个结论是不完整的**：核实的是发牌*顺序算法*对不对，没有核实动画*到底有没有真的播出来*，这是两件不同的事，后者才是用户真正在说的问题。

**根因（真实的竞态条件，靠证据定位，不是猜的）**：`GameTable.jsx` 里原本有个 `justDealt` 变量，纯粹从一个 ref（`prevPhaseRef`）派生，本质是"单次渲染脉冲"——只在 `prevPhaseRef` 还没追上 `gameState.phase` 的那一次渲染里等于 `true`。问题是：负责把 `prevPhaseRef.current` 更新到最新值的 effect，和负责触发 `setHeroRevealed(false)`（从而让 `dealing` 状态变 true）的 effect，在同一次 mount 的 effect 刷新周期里，按组件里声明的先后顺序**同步依次执行**——前一个 effect 先把 `prevPhaseRef` 追平，后一个 effect 才触发的那次重渲染里，`justDealt` 早就已经变回 `false` 了。也就是说，浏览器从来没有机会真正画出 `justDealt=true` 那一帧——用它控制的公共牌 `card-deal`、hero 自己的初始扣牌 `card-deal`、对手座位飞入 `deal-in` 三处动画，全部被无声跳过，外观上直接是"东西已经在那了"，不报错也不卡住，肉眼很难第一时间发现。

对照组也证实了这一点：对手手牌的 reveal 动画一直是好的，因为它用的是另一个信号——`dealing`（`=!heroRevealed`），一个真正持续存在、跨越整个发牌窗口的 state，而不是像 `justDealt` 那样的一次性脉冲。

**验证方法**：光看代码不够可靠（人眼容易被"逻辑看起来没问题"骗过去），改用真实证据——临时把 `DEAL_STEP`（每张牌之间的间隔）从 0.07 秒调到 4 秒，把整个发牌过程拉长到接近 1 分钟，刷新页面后立刻用 Playwright 读 DOM 的 `getComputedStyle(el).animationName`：公共牌、hero 扣牌、对手座位的 class 里根本没有 `card-deal`/`deal-in`，`animationName` 全是 `none`——实锤。

**选择**：整个删掉 `justDealt` 和它专属的 `prevPhaseRef`，三处动画统一改用已经验证可靠的 `dealing` 状态（新增具名变量 `const dealing = !heroRevealed`，与 `PlayerSeat` 的 `dealing` prop 同名同源）。这是"把还在用旧机制的角落，统一到已经验证 work 的机制上"，不是引入新概念，改动面小、风险低。

**教训**：这次的误判提醒了一件事——核实"规则/算法对不对"和核实"效果有没有真的呈现出来"是两个独立的检查，都要做，不能因为前者通过了就假设后者也没问题。以后遇到"动画/效果好像没生效"这类反馈，第一步应该是用类似的方式（拉长时间窗口 + 读真实 DOM 状态）直接验证效果是否触发，而不是只重新读一遍逻辑代码。

### 牌背改经典蓝格纹 + 账本盈亏合计 + 中文标签 + 撤销结算自动跳转（第十四轮，2026-07-19）

**背景**：用户连续反馈四个点。(1) 牌背用了跟桌布同色系的深绿+金色，用户反馈"不像市面上的扑克牌"，明确要蓝色格纹款。(2) 账本只有"当前"筹码，用户问"当前是什么意思，是不是应该有个最终盈亏"。(3) SB/BB/D 英文缩写希望换成中文"小盲/大盲/庄家"。(4) 亲测发现摊牌结算弹窗即使自己什么都没点，15 秒后也会自动开下一局——跟"必须每个人都点'我知道了'才能继续"的要求矛盾。

**排查发现**：结算的"15 秒自动跳转"不是本轮引入的新代码，是 `server/index.js` 里一段更早就存在的兜底定时器（`settlementFallbacks`），设计初衷是防止有人掉线或卡住导致房间永远卡在结算页——但它是无条件触发的，不管有没有人点确认都会在 15 秒后强制推进，直接绕过了"全员确认"这个产品要求，两者互相矛盾。同时发现结算弹窗上"等待其他人确认（X/Y）"这个数字其实是客户端本地伪造的（`iAmReady ? 1 : 0`，只反映"我自己点没点"），从来没有从服务端同步过其他人的真实确认进度，撤销自动跳转之后这个数字必须是真的，否则用户没法判断到底还要等谁。

**选择**：
1. `.c-back` 从"深绿+金色，低对比度菱格材质感"改成"深蓝渐变+白色格纹，可见对比度的经典款"，白色描边——直接对标市面常见的蓝色扑克牌背，不再往桌布同色系上靠。
2. `LedgerModal.jsx` 新增"盈亏"列，公式 = 当前 − 初始 − 已借（已借的筹码是借来的，不算真赢的，必须扣掉才是真实净输赢），正负用绿/红两色区分，说明文案同步写清楚公式。
3. `PlayerSeat.jsx` 的位置角标从 D/SB/BB 改成中文"庄家/小盲/大盲"，`.pos-badge` 相应加宽。
4. **删掉 15 秒兜底定时器**，结算弹窗不再有任何自动推进——只有两条路能进入下一手：全员真实点击"我知道了"，或者掉线玩家被移出待确认名单之后剩下的人已经都确认了（原有的"掉线不卡死其他人"这条保留，只是不再额外加一个无条件的时间兜底）。`RoomManager.js` 新增 `getSettlementProgress()`，`server/index.js` 在每次有人确认或掉线时把真实的 `{readyCount,totalCount}` 广播给全房间（新增 `game:settlement-progress` 事件），客户端不再自己伪造这个数字。
5. 顺带处理用户"头像和手牌会不会有点大"的顾虑：默认（非密集）尺寸头像 40→36px、对手常驻手牌 28×40→24×34px，密集桌（7-9人）同步收紧到 31px/19×27px，缓解玩家变多时的拥挤感。

**Trade-off（需要用户知情）**：删掉 15 秒兜底定时器意味着——如果某个玩家的确认请求因为网络问题真的没送达服务器（不是他们不想点，而是请求丢了），房间会一直卡在结算页等不到头，不像之前那样最多卡 15 秒就会自动放行。这是用户本轮明确要求的取舍（"必须每个人都点关闭才可以"），但要向用户说明这个代价，让其判断是否需要另一种更温和的兜底（比如卡住时房主可以手动强制推进），而不是完全没有任何安全网。

**验证**：服务端单元测试 75/75 通过（删掉定时器没有破坏任何既有断言）；`?states=` 自检画廊截图确认牌背/中文角标/账本盈亏列视觉效果符合预期。结算无自动跳转 + 真实确认进度这两条依赖真实多客户端 socket 交互，本地自检画廊无法覆盖，留待用户真机联机验证。

### 每局开局发牌动画（用户实测反馈第四轮，2026-07-18）

**背景**：task 9.9 落地的 `deal-in` 只在 `prevPhaseRef.current === null` 时触发——按注释"fires exactly on first mount"是有意为之，但实际效果是发牌动效只在整个会话的第一手牌播放过一次，从第二手开始就没有了。用户要求"每一局都要有"，且要按真实发牌顺序：按座位顺序发牌（从小盲开始）→ 亮出自己的手牌，正式开始这一局。

**决策**：
1. **修复 `justDealt` 只触发一次的问题**：判断条件从 `prevPhaseRef.current === null`（只在组件首次挂载时成立）改成 `prevPhaseRef.current !== 'preflop'`（任何一次"进入 preflop"的转变都成立——包括每一手新牌开局），配合已有的 `prevPhaseRef` 更新逻辑，改一行判断就让动效在每一局都重新播放
2. **发牌顺序 = 真实规则**：新增 `sbFirstOrder()`，以 `gameState.players`（座位固定顺序，不受 `getOrderedPlayers` 为 UI 重排的影响）为基础，从小盲位置开始轮转排序；每人 2 张牌按"轮次×人数 + 座位序号"算 stagger 延迟，等价于真实发牌员一圈一张、发两圈
3. **对手也有牌可看**：新增每个对手座位在发牌期间显示 2 张背面小卡（复用 `.reveal` 的定位槽位，摊牌揭牌和发牌动效互斥不会撞），按上面的顺序 stagger 弹入，营造"每个人都拿到牌了"的真实感；发牌流程结束后统一淡出/消失
4. **英雄手牌先背面落地、再翻开**：英雄的两张牌也先以背面按同样的发牌顺序 stagger 弹入，等全桌的牌都"发完"（`totalDealTime` = 按最后一张牌的延迟 + 单张落地动效时长算出来，不是拍的常量）之后，再统一翻面（复用已有的 `flip-reveal` 3D 翻面动效），翻牌那一刻就是"正式开始这一局"的信号
5. **踩过的坑**：第一版把"翻面倒计时"的 `useEffect` 依赖写成了 `[justDealt]`——但 `justDealt` 本质是"只在某一次 render 为 true 的脉冲"，下一次 render 就变回 false，依赖数组一变化就会立刻触发 effect 的 cleanup，把刚设的 `setTimeout` 取消掉，导致牌永远翻不回正面。改成依赖 `[gameState.phase]`（在整个 preflop 下注轮内这个值是稳定的字符串，不会跟着每次 render 抖动）才对：只在"刚进入 preflop"这一次真正触发，不会被同一 phase 内的后续 render 提前取消

### 发牌动画补全：公共牌也要先扣着发下来，到点再翻开（用户实测反馈第十一轮，2026-07-19）

**背景**：上面第四轮做的发牌动画只覆盖了玩家手牌（英雄+对手），公共牌区域在翻牌前一直是空的虚线占位框，直到翻牌圈/转牌圈/河牌圈实际到来时才凭空"变出"卡牌（配 `flip-reveal` 动效）。用户反馈：应该在给玩家发完手牌之后，公共牌也要用同一套发牌动效摆上桌——只是扣着（背面），等真正轮到那条街要开牌时，再把对应的牌翻面。这更符合真实牌局的观感：庄家一次性把这一手要用到的牌都摆好，该翻的时候才翻，而不是每条街都临时"变"出牌来。

**决策**：
1. **发牌顺序追加一段**：原来"手牌发完 → 英雄翻面"的两段式，改成"手牌发完 → 5 张公共牌扣着摆上（新增）→ 英雄翻面"三段式。公共牌沿用手牌那套 `DEAL_STEP` 间隔的 stagger 节奏，紧接在最后一张手牌落地之后逐张摆出，`totalDealTime`（决定英雄何时翻面）相应顺延，把公共牌摆牌这段时间也算进去。
2. **摆牌之后怎么"扣着"**：公共牌槽位从"没数据就画虚线空框（`.c-empty`）"改成"没数据但已经发过牌 → 画一张背面卡牌（复用 `Card` 组件的 `faceDown`）"；只有在这一手还没开始发牌（`waiting`/刚进房间还没 `justDealt` 过）时才保留虚线空框——否则每一局最初进入时会先闪一下背面牌，不自然。
3. **翻牌时刻沿用现成的 `flip-reveal`**：某条街真正揭晓（`gameState.communityCards[i]` 从没有变成有值）时，槽位从"背面卡牌"切换成"正面卡牌 + `flip-reveal`"——`flip-reveal` 本身的关键帧是从 `rotateY(-90deg)` 转到 `0deg`（见 task 9.9 落地时写的 `flipIn`），本来就是"侧面立起转正面"的视觉，只是过去应用在"从虚线空框凭空出现"上不够真实；现在改成应用在"从一张已经躺在原地的背面卡牌"上，才是名副其实的翻牌动效，这条 CSS 关键帧完全不用改，问题只在于"翻牌之前那张牌在不在"。

**验证**：Playwright 截图逐帧检查"手牌发完瞬间"确认 5 个公共牌槽位已经是背面卡牌（不再是虚线框）；截图检查翻牌圈/转牌圈/河牌圈揭晓瞬间确认对应槽位从背面变正面、其余槽位仍是背面；确认发牌总时长（英雄翻面时机）相应顺延、没有跟对手发牌动效撞在一起。

### 大厅断线宽限期：修复"房间不存在"的根因（用户实测反馈第七轮，2026-07-19）

**背景**：用户反馈"房主创建房间、把链接发给朋友，朋友打开时经常提示'房间不存在'，要重试几次才行"。排查后定位到根因，且顺带修了两个相关的连带 bug。

**根因**：`server/index.js` 的 `disconnect` 处理器过去是这样的：任何 socket 断开，不管房间处于什么状态，立刻 `rooms.leave(myPlayerId)`——如果这个玩家是房间里唯一的人（房主刚建完房、朋友还没加入进来，正好是这个状态），房间会被直接删除（`RoomManager.leave()`：`players.length===0` 就 `this.rooms.delete(code)`）。而"房主创建房间后，切到微信/短信把链接发出去"这个动作，在移动端浏览器上很容易导致标签页被系统挂起、WebSocket 连接短暂断开——即便 socket.io 客户端几秒后会自动重连，服务端在断开的那一刻就已经把房间删没了。朋友点链接进来时，房间已经不存在，房主自己可能都没察觉（页面还停在大厅旧状态，重连后也没有任何机制告诉服务端"我回来了、别删我"）。房主往往通过"重新建一个房间再发一次"来解决，这正好对应用户描述的"重试几次才行"。

**决策**：
1. **大厅阶段断线加宽限期**：只有 `room.status==='waiting'`（大厅/未开局）时才延后处理断线——不立即 `rooms.leave()`，而是设一个定时器，到期时检查这个玩家是否已经用新 socket 重新关联（`socketId` 是否还指向这个已经断开的旧连接），没有才真正移除。游戏进行中的断线依然立即处理（自动 fold + 移除），不受影响，MVP 阶段"游戏中断线不支持重连"的既有决策不变。宽限期时长最初定为 30 秒，用户反馈"可能不够"后改为 **120 秒**（`GRACE_PERIOD_MS`），给切 App 分享链接/临时被打断留更宽松的容错
2. **重连超时的显式报错**：之前如果玩家超过宽限期才回来（房间已经被真正移除），`room:sync` 只是静默返回，客户端会卡在一个数据早已过期的大厅界面上，自己毫无察觉。新增 `room:gone` socket 事件：`room:sync` 找不到房间时显式告知客户端，`RoomPage.jsx` 收到后弹出"重新连接超时，房间已失效，请重新创建或加入"的 toast，2.5 秒后自动跳回首页（跟已有的 `room:kicked` 处理方式一致）
2. **修复连带 bug——`room:sync` 从未真正"重新连接"过**：原来的 `room:sync` 只是读一下房间状态发回去，从来没有调用过 `room.updateSocket()`（更新玩家的 socketId）、`socket.join(room.code)`（重新加入 socket.io 房间广播列表）、也没有设置这个连接自己的 `myPlayerId`。也就是说就算断线重连的宽限期加上了，重连后的客户端也收不到后续广播、它自己下次断线也不会被正确处理——这是一个更早就存在但没被注意到的 bug（正常整页刷新时，旧连接已经在刷新前断开触发了立即删除，恰好掩盖了这个问题；只有"标签页不刷新、底层 socket 断了又连上"这种情况才会暴露）。修复：`room:sync` 现在会完整地重新执行这几步关联
3. **客户端配合**：`RoomPage.jsx` 的 `room:sync` 调用从"只在组件挂载时发一次"改成"每次 socket 触发 `connect` 事件都发一次"（挂载时的首次连接和之后任何一次重连都会触发 `connect` 事件），否则移动端标签页被挂起又恢复时，客户端不会主动告诉服务端"我回来了"
4. **验证方式**：Playwright 直接操作页面里暴露的 socket 实例（`window.__vrSocket`，新增的调试钩子）强制断开/重连，而不是用浏览器级别的离线模拟——`context.setOffline()`/CDP `Network.emulateNetworkConditions` 在这个沙盒环境里会影响到同一个浏览器进程里其他不相关页面的网络，不适合模拟"只有某一个玩家断线"的场景。真实复现了完整场景：房主断线 3 秒（房间未删除）→ 朋友在房主离线期间成功加入 → 房主重连后正确看到朋友已经在大厅里

1. **骨架：椭圆 rail → 贴边双栏**。`seatPositions()`/`spectatorSeatPositions()` 的参数方程整体废弃，改为：英雄固定屏幕底部中央不变；其余对手按人数尽量均分成左右两栏（如 8 个对手 = 左4右4，7 个 = 左4右3），每栏从上往下固定行距堆叠，越靠下越接近英雄；中间留一条竖直空白通道给底池/公共牌/桌名信息，不放座位。下注筹码/气泡"飞向中心"的方向简化为"水平飞向中间通道"，`betChipStyle` 不再需要椭圆切线角度计算。`.table-canvas` 的 `scaleX`/`scaleY` 独立缩放 + 内容层 `csx`/`csy` 反向抵消这套机制（第九/十二轮）继续沿用——只是作用对象从"椭圆 rail 坐标"换成"双栏坐标"，机制本身不受影响。座位数容量（2-9 人，9 人=8 对手+我，房满即拒绝加入）维持既有决策不变。

3. **状态气泡常驻化，`.fold-tag`/`.allin-tag` 下线**。`.action-bubble` 从"1.6 秒后淡出"改为"常驻，直到该玩家下一次行动或进入下一街清空"——过牌/跟注/加注/ALL IN/弃牌统一走这一套常驻气泡（弹出动画保留，只是不再消失）。现有单独的 `.fold-tag`（弃牌）、`.allin-tag`（ALL IN）底部灰字下线，并入常驻气泡机制，避免同一含义有两套互相冲突的展示。下注筹码浮标（`.bet-chip`）不受影响，继续保留，金额随下注变化更新。

4. **行动指示：呼吸光环动画 → 静态描边 + 读秒遮罩**。`.is-active .avatar` 去掉 `activePulse` 脉冲动画，改为静态金色描边（不再一明一暗）——静态描边负责"扫一眼定位轮到谁"，跟下面的读秒遮罩分工不重叠，避免两者抢注意力。新增读秒遮罩：轮到某玩家时，其头像照片区盖一层半透明遮罩，遮罩上叠加正向读秒数字（"已思考 N 秒"）。**不做圆环/扇形消退动画**（参考 App 是倒计时消退环，用户明确不要这种、只要遮罩+数字）。计时逻辑**完全客户端本地实现**：每个客户端在自己观察到 `gameState.actionPlayerId` 变化时本地清零重新计时，不设上限、不做超时惩罚，不需要任何服务端改动或状态同步——各客户端因网络延迟导致读数有毫秒级差异是可接受的，这是氛围展示，不是权威计时，不影响游戏逻辑正确性。

5. **新增互动：拍一拍**。点击其他玩家的座位卡片 → 客户端发送轻量 socket 事件（`player:poke`，`{fromId, targetId}`）→ 服务端广播给全房间（`player:poked`，`{fromId, targetId}`）→ 所有客户端在目标玩家的卡片上播放一次抖动/摇晃动画（CSS keyframe，约 0.4-0.6 秒），并弹出一个复用气泡组件"瞬时版"的"戳了戳"小气泡（区别于第 3 点的"常驻"气泡，这个仍是短暂淡出——语义上是即时社交反馈，不是本轮牌局状态）。**防刷**：服务端对同一 `fromId→targetId` 组合加 2 秒冷却，冷却期内的重复请求直接忽略（不报错，静默丢弃）；不允许 `fromId===targetId`。弃牌/旁观状态的玩家卡片同样可以被拍（纯社交动作，跟牌局状态无关）。不做持久化的拍一拍消息记录——项目目前没有房间聊天/消息记录这层基建，不为这一个小功能单独新建。

6. **视觉打磨项**（具体像素值走项目既有的 `states-preview.html` 预览页流程确认后再落地，这里只定方向，不定死数字）：
   - 英雄手牌尺寸降一档（`md`→`sm`）——参考截图里对手/系统预留的手牌尺寸很小，用户认为这类信息本身不重要，只是发牌动画那一刻的仪式感需要；英雄自己的手牌因为是核心决策依据，保留现有更大尺寸不变
   - 底池/街道指示文字对比度不足，需调整颜色/透明度提升可读性（具体数值预览页确认）
   - 桌面绿色从当前的纯渐变，改为带轻微噪点纹理的"绒面"质感，视觉目标是模拟真实牌桌毛毡材质——用 `frontend-design` skill 出图对比，不凭空猜测参数

**受影响/superseded 的既有决策**：第五轮（椭圆放大+头像贴边线）、第六轮（头像统一贴边线去掉例外）中"椭圆参数方程"相关的具体数值（`cx/cy/rx/ry`、`MIN_OPPONENT_Y`/`CARDS_SIDE_BELOW_Y` 等）随骨架整体替换一并作废，仅作历史记录保留在原处，不删除；第九轮、第十二轮里"`.table-canvas` 非等比缩放 + 内容层反向抵消变形"这套缩放**机制**本身不受影响、继续沿用，只是不再服务于椭圆坐标而是双栏坐标；v3 评审"姓名由头像字符承担、去掉昵称"的决定本轮推翻，昵称恢复显示。

**待预览页验证**（不在本轮文字确认范围内，需要出图后用户确认才算数）：双栏行距、卡片精确尺寸、9 人密集桌降级规则、昵称截断长度、噪点纹理参数、读秒遮罩视觉细节。

### 房主切 App 再切回来，重连后变成"全新受邀者"（用户反馈第二十轮，2026-07-20）

**背景**：用户反馈"房主创建房间后，切到微信或其他 App 再切回浏览器（移动端很容易发生，tab 经常被系统回收），连接会断开；用同一个邀请链接重新进入，系统会把我当成一个全新的普通受邀者放进房间，而不是恢复我原来的房主身份和状态"，且"几乎一断线就会这样"。

**根因**：第七轮已经修好了"socket 断线又重连、但页面/React 没重新加载"这一种重连路径——`RoomPage.jsx` 挂载时和每次 `socket.on('connect', ...)` 都会发 `room:sync`，服务端也正确地把新 socketId 关联回原玩家。但移动端"切 App 再切回来"最常见的情况不是这一种，而是系统直接把后台标签页整个回收，切回来时是**全新的页面加载**——`App.jsx` 的 `room` state（进而 `RoomPage` 是否挂载）在每次冷启动时都从 `null` 开始，且 `App.jsx` 的初始化逻辑只看 URL（`/room/:code` 路径或 `?room=` 参数），从未读取过 `localStorage` 里已经存好的 `vr_playerId`/`vr_roomCode`（`HomePage.jsx` 在 `room:joined` 时写入，但只在 `getPlayerId()` 里当"生成/复用一个匿名设备 ID"读出来，从未用它判断"这其实是一次会话恢复"）。于是每次冷启动都无差别地渲染 `HomePage` 走人工加入表单，最终触发的是 `room:join`（`RoomManager.addPlayer`），而不是本该走的 `room:sync`：
- 如果这时还在 120 秒宽限期内，`addPlayer` 发现这个 `playerId` 还在房间里，直接返回"已在房间内"报错，卡死，没有任何自动恢复路径；
- 如果宽限期已过（`removePlayer` 已把 `hostId` 转给 `players[0]`、并把这个 id 从名单里删掉），`addPlayer` 会把这个复用的 `playerId` 当成**全新玩家**塞进去——筹码重置为初始值、且早已不是房主——这正是用户看到的现象。

第七轮的验证方式（用 `window.__vrSocket` 在同一页面内强制断开重连）恰好完全绕开了"整页冷启动"这条路径，所以这个缺口当时没有被测出来。

**决策**：这是一个客户端会话恢复缺失的问题，不是服务端的问题——服务端 `room:sync` 的重新关联逻辑（第七轮已修）本身是完备的，只是从来没被冷启动路径调用过。修法是让 `App.jsx` 在决定渲染 `HomePage` 还是 `RoomPage` 之前，先检查是否存在一个"值得恢复"的本地会话，值得恢复就直接进 `RoomPage`（复用 `RoomPage` 已有的、挂载即发 `room:sync` 的逻辑，不新增一条平行的恢复路径）：
1. **判定"值得恢复"**：本地存在 `vr_playerId` + `vr_roomCode`，且（当前 URL 没有指定房间号 `/`、`/room/CODE` 两种路径都算，只要 URL 里的房间号为空或与 `vr_roomCode` 相同）——不匹配（点了一个别的房间的邀请链接）时不触发恢复，走原来的人工加入表单，这种情况下"沿用同一个匿名 playerId 去加入一个新房间"是合理的、不是这次要修的场景
2. **恢复路径 = 直接挂载 `RoomPage`**：不新增单独的"恢复中"加载态或专门的 socket 事件，直接把 `room` state 设成 `{code: vr_roomCode, playerId: vr_playerId}` 挂载 `RoomPage`——`RoomPage` 已有的 `room:sync` on-mount/on-connect 逻辑原样生效；服务端如果在宽限期内找到房间，正常 `room:state`/`game:state` 回来，身份（含房主）原样恢复；服务端如果宽限期已过找不到房间，已有的 `room:gone` → toast → `onLeave` 路径原样生效，退回首页
3. **会话结束时清理本地存储**：`room:gone`、被踢 (`room:kicked`)、主动退出 (`onExit`) 这三条路径最终都会调用 `App.jsx` 的 `handleLeave`——之前只重置 `room` state 和 URL，没有清 `localStorage`。现在一并清掉 `vr_roomCode`（`vr_playerId` 保留，作为跨房间延续的匿名设备身份，之前的行为不变），否则"确认已经离开/房间已失效"之后，下次冷启动还会再尝试恢复同一个已经不存在的会话
4. **没有触碰的部分**：`RoomManager.removePlayer()` 里宽限期真正过期后 `hostId` 转给 `players[0]` 的单向转移逻辑保持不变——一旦真的超过 120 秒未回来，视为这位玩家确实离开，房主转移给别人是合理的产品行为；本次修复的范围是"让原本该走的重连路径真正被走到"，不是"允许过期后仍找回房主身份"，避免过度设计一个没有明确产品诉求的"迟到房主抢回身份"机制

**验证方式**：Playwright 真实浏览器测试，用 `page.reload()`（模拟系统回收标签页后的整页冷启动，而不是第七轮用过的同页面 socket 级断线）复现：房主建房 → reload → 断言直接看到大厅（不经过加入表单）、`hostId` 仍是自己、看得到"开始游戏"按钮；同时覆盖"reload 发生在宽限期过期之后"（用 `window.__vrSocket` 提前把这个玩家从房间移除模拟宽限期耗尽）应该退回首页而不是死循环；以及"点了另一个房间的邀请链接，本地却有别的房间的旧会话"不应该被误判为恢复。

### 第十九轮上线后真机实测：两处重叠/裁切 bug + 去掉椭圆装饰（2026-07-20）

**背景**：第十九轮合并部署后，用户在真机（iPhone，2 人局）上截图反馈：①左列座位（顶排）的暗牌图标被屏幕左边缘裁掉一半；②英雄自己的下注气泡、姓名、筹码、头像读秒挤在一起糊成一团看不清；③背景那个绿色圆圈跟新的贴边双栏座位完全对不上号，人少时显得"一个座位孤零零地贴在一个巨大空圆圈角落"。

**根因**：
1. `cardsSide` 方向判断反了——`GameTable.jsx` 里顶排座位把暗牌/气泡推向"所在列的外侧"（左列推左、右列推右），本意是避免顶部裁切，但外侧对左列来说就是屏幕边缘本身，直接把暗牌推出可视区域。
2. 英雄小头像座位的 `y` 坐标（585）离画布底边（610）只有 25px，完全没给下面 `.hero-section`（姓名+筹码+大手牌）留出空间——旧椭圆算法算出来的英雄位置在 y≈395，留了 215px，这次改版重写坐标时没有照这个余量来，也没有一个测试场景真正同时渲染"英雄是行动方（带读秒）+ 英雄有下注（带气泡）"这个组合，所以没被现有验证发现。
3. 背景圆圈（`.table-oval` 的 `border-radius:50%` + 描边阴影）是专门按已废弃的椭圆座位尺寸画的装饰，座位改成贴边两栏后这个形状已经没有存在意义。

**决策**：
1. `cardsSide` 方向改为推向中间通道（有空间的一侧），而不是所在列的外侧。
2. 英雄小头像 `y` 坐标重新按实测留出的安全余量确定（不再是拍脑袋的 585），保证跟 `.hero-section` 不重叠——具体数值走 Playwright 真实渲染量出来，不手算。
3. **去掉圆圈装饰**：`.table-oval` 不再画圆（去掉 `border-radius:50%` 和描边/阴影），绒面质感背景（第十九轮 Task 8 做的噪点+渐变）移到 `.table-canvas` 整个画布上，覆盖两栏之间的整个牌桌区域，不再是一个跟座位对不上号的几何形状——跟用户最早参考的截图（纯色绒面背景+两栏座位+中间底池）方向一致。

**验证**：Playwright 真实渲染量 bounding box（不是手算），覆盖 2 人局（复现用户截图场景）+ 9 人满桌两种极端，确认暗牌不再出画布、英雄信息区不再重叠；上述两条断言固化成永久 e2e 回归测试（`e2e/game.spec.js` "真机实测回归"），配套新增 `fixtures.js` "英雄行动中且有下注" 状态（此前没有任何 fixture 同时覆盖"英雄是行动方+英雄有下注"这个组合）。

**同一轮用户追加的两项调整**：
1. **下注气泡改版**：从"深色文字胶囊+SVG 尾巴"（第十五/十六/十八轮反复调整过尾巴方向/形状）改成"筹码图标（SVG 绘制的同心圆，模拟真实筹码边缘）+ 金额数字"，不再需要尾巴指向座位这套机制，`betChipStyle()` 里 `--tail-deg` 的计算一并删除。用户明确要"参考截图那种实物筹码样式"。
2. **英雄小头像卡片恢复筹码 footer**：`.player-slot--hero` 之前单独隐藏 `.stack-chip-footer`（避免跟下方 `.hero-section` 的筹码数字重复），用户要求"我的头像布局要跟其他玩家的布局一样，在长框里面"——视觉一致性优先于避免数字重复，改回跟其他座位同一张"头像+footer"卡片样式。

### 同一轮追加：暗牌/筹码/公共牌互相遮挡 + 座位间距按人数动态调整（2026-07-20）

**背景**：上面几项改完后用户继续反馈：①对手暗牌挡住了自己的下注筹码显示；②暗牌也会挡住桌面中间的公共牌（多人/密集桌场景更明显）；③头像下方筹码数字不用加粗，且要保证不超出卡片边框；④座位间距应该按人数动态调整——人少时更松，人多时才紧一点（现状固定间距不管人数多少都一样紧）。

**根因**：把"暗牌永远推向中间"从"仅顶排座位"扩大到"每一排座位"之后（解决了暗牌被裁切/被相邻行遮挡的问题），新问题是暗牌、下注筹码、公共牌三者都在往"中间通道"这同一块窄空间里挤——暗牌离座位不够近、筹码飞的不够远、公共牌区域纵向没有给最下面一排座位让出余量，三者在人多的密集桌上必然互相撞上。

**决策**：
1. **暗牌尺寸再缩小、离座位更近**：`.c-xs`（暗牌专用尺寸，不影响其他地方的牌）从 24×34 缩到 17×24（密集桌从 19×27 缩到 14×20），`sideStyle()` 的贴边间距从 `calc(100%+6px)` 收紧到 `calc(100%+3px)`——暗牌离座位更近、占用的"往中间"空间更少，给筹码和公共牌让出余量。
2. **下注筹码偏移量加大**：`BET_CHIP_OFFSET` 从 40 提到 92，让筹码飞得比暗牌更远、明确落在暗牌外侧，不再抢同一块空间。数值经 Playwright 实测调出来，不是手算。
3. **公共牌区域纵向范围收紧**：`.table-oval` 的 `bottom` 从 180px 调到 200px，公共牌区整体略微上移，跟最下面一排座位的暗牌之间留出实测确认的安全间距（9 人密集桌场景，最紧的情况）。
4. **座位间距按人数动态**：新增 `rowPitchFor(rowsPerColumn)`——单栏 ≤2 排时间距 130px，3 排时 100px，4 排（7-9 人满桌）时 76px（原来的固定值，保留作为最密情况的下限）。人少时明显更松，只有真正坐满时才收紧到当前这个密度。
5. **筹码数字防溢出**：`.stack-chip-footer` 字重从 700 改成 400（不加粗）；加 `overflow:hidden`/`white-space:nowrap`/`text-overflow:ellipsis`，大额筹码（如 ¥15,989）超出卡片宽度时截断显示省略号，不会溢出卡片圆角边界。

### 单挑大小盲分配错误 + 摊牌前对手暗牌下线 + 交互/可读性打磨（2026-07-21）

**背景**：用户实际联机测试时发现"翻牌前自己还没操作，头像旁边就已经有一个¥20下注标识"，追问后确认这不是操作产生的。同时反馈：预设加注按钮"1/3池"跟"半池"有时显示成一样的数字、两个按钮同时被高亮；预设按钮、"等待其他玩家的行动"等文案偏小/对比度不足；行动方头像描边太粗太复杂；座位间距要留够、不要太挤；对手摊牌前的两张暗牌占地方又不传递信息，建议去掉，摊牌时才需要真的亮出来。

**根因（单挑大小盲）——真正的规则 bug，不是视觉问题**：`GameEngine.js` 的 `_assignBlinds()` 用的是 3 人以上环形桌的公式（`sbIdx=dealer+1, bbIdx=dealer+2`）。这个公式在**只有 2 人（单挑）时会退化出错**：`(dealer+2) % 2` 数学上等于 `dealer` 自己，也就是把庄家错误地摊派成大盲（该由庄家来的小盲反而摊派给了另一个人）。真实德州扑克规则里单挑时庄家应该是小盲、且翻牌前先手。这个 bug 一直存在，之前的单测只断言过"底池=大小盲之和"（`pot===300`），没断言过具体谁是哪个盲注，所以从未被测出来；`_endHand` 相关的两条集成测试里"p1 是庄+大盲、p2 是小盲先手"的注释其实是把这个 bug 的表现当成了正确行为写进了断言，一并修正。

**决策**：
1. `_assignBlinds()` 新增 `n===2` 分支：庄家＝小盲，另一位＝大盲；三人以上桌逻辑不变。
2. 翻牌前的行动顺序（`actionIndex` 初始化）同步新增 `n===2` 分支：单挑时从庄家（小盲）开始行动，而不是环形桌规则的"大盲左手边"。
3. **摊牌前对手暗牌整体下线**：`PlayerSeat.jsx` 里 `hasCards && !isMe && !folded && !isShowdown` 这段常驻暗牌渲染直接删掉——这两张牌摊牌前永远是背面、不传递任何信息，纯粹占地又跟下注筹码/公共牌抢中间过道空间（上一轮才刚花了不少功夫去解决这三者互相遮挡的问题）。去掉后，`cardsSide`/`sideStyle` 这套"贴边"逻辑不再需要覆盖这个场景，只保留服务摊牌揭示（`isShowdown` 分支不受影响，逻辑本来就是独立的两个条件分支）。
4. **行动方描边简化**：`.is-active .avatar-card` 从"3px 实心描边 + 两层 box-shadow 叠加"简化成更克制的单层样式。
5. **加注预设"撞车"修复**：某档预设的原始计算值低于本手最低加注额时，会被 clamp 顶到跟别的档位一样的数字，此时两个按钮同时高亮，看着像坏了。改为：预设值被 clamp 改写时，该档显示为置灰/不可用样式，跟未被 clamp 的档位区分开，不会两个同时点亮。
6. **可读性/触达性打磨**：预设按钮 26px/10px→38px/12px；`.waiting-text`（"等待其他玩家的行动"）颜色从 `#203820`（背景约 `#050D06`，实测对比度接近 0）改成 `#9FB0AC`（跟其他次要信息同一套灰绿色系），字号 9→11px；读秒数字 11px→15px，加深投影。
7. **座位间距**：暗牌下线后中间过道天然宽裕了不少，实测（2 人局、9 人满桌两个极端）现有 `rowPitchFor`（单栏≤2排130px/3排100px/4排76px）已经不再拥挤，未进一步调整数值——如果后续实机反馈仍然觉得挤，再回来加宽。

**验证**：新增服务端单测覆盖单挑大小盲分配（含庄家非 0 号座位的情况）；受影响的既有集成测试（原先断言"p2 先手"）改为断言正确的"p1（庄+小盲）先手"，配套调整弃牌/断线的角色；Playwright 实测确认暗牌下线后中间过道不再拥挤、预设按钮不再撞车高亮、对比度提升后真实可读；用真实两人对局（非 fixture）截图确认庄家正确显示"庄家+小盲"、对手正确显示"大盲"。

**踩坑记录（沙盒环境）**：这一轮验证时反复遇到"跑全量测试时随机有几个多人 Playwright 用例失败，单独重跑又全过"的情况。排查后确认是这个沙盒环境的已知限制（design.md 更早的踩坑记录里提过"单进程里第 3 个浏览器 page 容易卡死"）叠加了这次会话里恰好有另一个 Claude Code 窗口也在同时跑 Playwright、抢占同一台机器的资源/端口——不是这一轮代码改动引入的问题：把每个报错的用例单独、多次重跑，全部稳定通过；server 端 82/82 全绿。

### 下注显示统一为"分类+金额"气泡，去掉筹码 icon 样式（用户反馈，2026-07-21）

**背景**：用户截图指出场上同时存在两种下注提示样式——"筹码 icon + 数字"（`.bet-chip`）和"分类文字气泡"（`.action-bubble`，如"加注 ¥40"）。要求统一成后者一种样式，覆盖包括自己在内的所有玩家，并让分类更完整（问是否该有小盲/大盲），同时希望这个提示离头像更近、更醒目。

**决策**：
1. 整个删除 `.bet-chip` 相关代码（`GameTable.jsx` 里的 `BET_CHIP_OFFSET`/`betChipStyle`/两处渲染，`velvet.css` 里的 `.bet-chip`/`.bet-chip::before`/`chipPop` 动画），下注提示只保留 `.action-bubble` 一条路径。
2. 分类补全：原有的加注/跟注/过牌/弃牌/ALL IN 是通过"行动前后 diff"生成的，但翻牌前的小盲/大盲下注是服务端在任何 `actionPlayerId` 切换之前就已经下好的，diff 逻辑天然捕捉不到——新增一个"进入新一手翻牌前"的 effect，直接按 `isSB`/`isBB` 标志给对应玩家种上"小盲 ¥X`"/"大盲 ¥X`" 气泡，谁先真正行动时会用自己的真实动作气泡覆盖掉种子气泡。
3. **独立复核用户提出的分类列表**（用户明确要求不要无脑照抄，自己判断合理性）：小盲/大盲/加注/跟注/过牌/弃牌/ALL IN 这七类已经覆盖德州扑克翻牌前到摊牌的全部合法动作，没有遗漏；但发现一处不一致——`ALL IN` 原来只显示文字不带金额（`text = 'ALL IN'`），跟其余几类"分类+金额"的格式不统一，顺手改成 `ALL IN ¥{当前下注额}`（用总下注额，跟"加注"的口径一致，而不是"跟注"用的增量）。
4. `.action-bubble` 尺寸/字重加强以提升醒目度：`padding` 4px 10px→5px 12px，字号 11px→13px，字重 600→700。

**踩坑记录（根因已定位）**：把 `.action-bubble` 接到第 0 排座位专用的 `sideStyle()`（贴边定位，避免顶排内容顶出画布上边缘）之后，实测发现文字被挤成一团、宽度只有 26px。用 `getComputedStyle()` 排查后确认：`.action-bubble` 基础 CSS 类本身带着 `left:50%`（默认"贴头像正上方居中"用的），而 `sideStyle('left')` 内联样式原来只设置了 `{position, right, top, transform}`，没有覆盖 `left`——于是 `left:50%` 和内联的 `right:calc(100%+3px)` 同时生效，浏览器把元素当成"两条边都被约束"来算宽度，而不是"以 right 为锚点收缩自适应"，导致挤扁。对照 `.reveal`（摊牌揭示用的同一个 `sideStyle()`）从来没出过这个问题，正是因为 `.reveal` 的基础类完全没声明 `left`/`bottom`/`position`。修复：让 `sideStyle()` 每个分支都把 `left`/`right`/`top`/`bottom` 四个属性显式写全（用不到的设成 `auto`），不再依赖调用方所在 CSS 类当前是否留了冲突的默认值——这样以后接到任何基础类身上都不会有这类隐性冲突。

**验证**：Playwright 实测两人局翻牌前（小盲/大盲气泡文字与位置）、加注、ALL IN（确认气泡显示"ALL IN ¥X"且不裁切）；`?states=` 静态 fixture 网格逐个截图核对（第 0 排座位"陈美玲"气泡贴边定位后不再挤扁乱码，宽度恢复到跟居中气泡一致的 ~80px）；确认 `.bet-chip` 元素计数为 0；服务端 103/103 测试全绿。

### 游戏中途断线从"自动弃牌+移出房间"改为"暂停等待重连"（用户反馈第二十一轮，2026-07-20）

**背景**：完整设计见 `docs/superpowers/specs/2026-07-20-mid-game-reconnect-design.md`，这里只记关键决策，不重复展开。

**根因**：`room.status==='playing'` 期间的断线处理，从 MVP 早期起就是"游戏进行中不支持重连"这条明确设计决策的产物——玩家一断线，服务端立刻自动帮他弃掉当前这手牌，紧接着直接把这个人从房间彻底移除，完全没有宽限期，跟大厅阶段（`waiting` 状态）120 秒宽限期后才移除的行为不是一回事。这在真实使用中太粗暴：移动端网络抖一下、切一下 App，就可能直接把人踢出正在进行的牌局。

**决策**：把"自动弃牌 + 移出房间"这两个动作都去掉，改成暂停等待。`GameEngine` 本来就会拒绝非当前行动玩家发起的操作（`idx !== this.actionIndex` 守卫），所以断线玩家轮到他时，牌局会自然卡在那一步等待，不需要新增一个专门的"暂停状态"——去掉这两个有害行为本身就足够。三种解除卡住的方式：玩家自己重连、房主手动"帮TA弃牌"、或 5 分钟安全超时自动弃牌兜底（避免房主本人断线导致整桌死锁）。断线玩家在重连之前会被跳过、不发进新的一手（既不移除也不强行发进去），重连后自动重新计入下一手。**唯一**会真正丢失座位的情况是房主主动踢人或筹码归零出局——断线时长本身，不管多久，都不是移出房间的理由，这跟大厅阶段"120 秒宽限期后移除"的既有行为不同，是本轮刻意保留的差异点。

**范围收敛**：这一轮的客户端改动刻意做到最小（纯文字 toast + 一个普通按钮），没有触碰 `GameTable.jsx`/`PlayerSeat.jsx`/`velvet.css` 里的座位卡片视觉——因为并行分支正在重做这些组件，完整的视觉呈现（比如断线状态在座位卡片上的样式）推迟到那条分支合并之后再做。

**验证方式**：Playwright 真实浏览器测试，用页面暴露的调试钩子 `window.__vrSocket.disconnect()` / `.connect()` 模拟牌局进行中真实断线再重连；服务端另有集成测试覆盖，包括一个用假定时器验证 5 分钟安全超时确实会触发的用例。

### 真机截图追加反馈：气泡遮挡头像 + 位置标签挪到名字旁 + 头像/英雄结构统一 + 一批可读性细节（用户反馈，2026-07-21）

**背景**：用户发来一张真机截图，对手"加注 ¥40"的气泡糊在了自己的头像上（把 ¥940 都挡住了）；同一条消息里还带了一批细节反馈：气泡对比度不够亮；庄家/小盲/大盲的位置标签应该挪到名字旁边而不是贴在头像上；英雄自己的座位结构（名字位置）跟其他玩家不统一；英雄头像卡片偏窄；读秒数字想用橙红色而不是金色、且激活头像应该带呼吸动画跟读秒同频；背景纹理在手机上不够清楚、偏暗。用户明确要求"自己归类总结，有疑问再问"，且要求独立判断他给的分类是否合理（呼应上一轮的"不要无脑跟随"要求）。

**决策**：
1. **英雄名字位置**：先用 AskUserQuestion 问清楚——用户先选"反过来统一成头像在上"，随后改主意确认为"跟对手一致，名字在上"。实现：取消英雄小头像座位隐藏 `.seat-name` 的规则，`.hero-section` 里原来单独显示的"姓名"整块删掉（不再重复），姓名后缀"（我）"现在直接拼在 `PlayerSeat` 的 name 文本里（`isMe && '（我）'`），不再是 hero-section 专属的写法。
2. **气泡覆盖头像 —— 结构性修复而非补丁**：原来只有第 0 排座位的气泡走贴边定位（`bubbleSide`），其余排用默认的"贴头像正上方"定位——真机截图里出问题的这个座位其实也是第 0 排，说明问题不只是"要不要贴边"，而是贴边定位本身在真实对局（非静态 fixture）里没生效。改为**所有座位的气泡一律贴边定位**（`bubbleSide = cardsSide`，跟摊牌揭示牌用的方向一致），不再区分排数，逻辑更简单也更彻底。
3. **气泡覆盖头像的真正根因（比 3.4 的 CSS 修复更深一层）**：贴边定位在真实对局里仍然没生效，用 `getComputedStyle` + 检查内联 `style` 属性排查后发现——`actionBubbleIn`/`actionBubbleOut` 这两个入场/退场动画的每一帧关键帧都硬编码了 `transform:translate(-50%, ...)`（历史遗留：这俩动画是在"气泡永远居中贴在头像正上方"的年代写的，用关键帧顺带提供居中位移）。CSS 动画只要还在生效（`animation-fill-mode:both` 意味着动画结束后也永久生效，不会把控制权还给静态/内联值），就会**整个接管** `transform` 属性，内联样式设的 `translateY(-50%)`（贴边定位要用的垂直居中）完全不起作用，气泡因此被强行拉回水平居中，正好糊到头像上——这跟 30 号任务里修的"`left`/`right` 冲突"是两个独立的 bug，凑在一起才让气泡的位置/宽度都不对。**修复**：把入场/退场动画里的位移+缩放，从共用的 `transform` 属性挪到独立的 `scale`/`translate` CSS 属性（现代浏览器原生支持，二者与 `transform` 天然叠加而不是互相覆盖），`transform` 从此只由定位逻辑（居中或贴边）掌控，动画只负责"弹入"效果本身。副作用：默认居中的气泡（原来完全靠动画关键帧的 `translate(-50%)` 提供居中位移，没有自己的静态 transform）失去了居中效果，随手在 `.action-bubble` 基础类补回 `transform:translateX(-50%)` 静态值。
4. **位置标签挪到名字旁**：`.pos-badge` 从"绝对定位贴在头像右下角的角标"改成跟名字同一行的行内 pill（新增 `.seat-name-row` flex 容器包裹名字+标签），不再覆盖头像本身。
5. **对比度**：`.action-bubble` 文字色 `#F5E6A0`→`#FFF3C4`（更亮），边框透明度 .5→.75，背景略加深、加一圈淡金色外发光，整体更跳。
6. **读秒配色+呼吸动画**：`.think-overlay` 颜色金色 `#F5E6A0`→橙红 `#FF7A45`（金色在这张桌面上到处都是"正常态"的颜色，读秒需要一个更有紧迫感、不会被误认成"平常"的独立色系）；新增 `.is-active .avatar-card` 的 `activeBreathe` 呼吸动画（1s 周期 scale 1↔1.045，跟读秒每秒跳一次的节奏取相同周期，不做 JS 精确对频，一个同周期的 CSS 循环已经足够读出"合拍"）。

### 摊牌/结算呈现重新设计：弃牌结束 vs 真摊牌分开、结算弹窗不再挡住手牌（用户反馈，2026-07-21）

**背景**：用户反馈摊牌体验一直有问题——"总是看不到各家的牌""弹窗好像会挡住自己的牌"；同时提出结算总结应该突出"赢家靠什么牌赢的"，并且弃牌结束（没有真实摊牌）跟真摊牌应该在样式和内容上区分开来，不能用同一套呈现。这是一次"怎么设计"的探讨，我先给了方向性建议（弃牌结束用轻量提示、真摊牌给桌面留时间+挪开弹窗位置），用户确认后动手实现。

**根因排查**（读代码确认，非猜测）：
1. `.settlement-sheet` 是从底部升起、`max-height:62%` 的常驻底部抽屉，`game:showdown` 事件一到就跟摊牌揭示牌同时出现——高度直接盖过了英雄自己固定在画布底部（`bottom:28px`）的手牌，以及下半区好几排对手的揭示牌，用户根本来不及看。
2. 服务端其实已经在区分"弃牌结束"（`handName:'对手全部弃牌'`，`contenders.length===1`）和"真摊牌"（`handName: 手牌类型描述`），只是客户端两种情况用的是同一套 `SettlementModal`，没把这个区分体现在呈现上。

**决策**：
1. **拆成两个组件**：`_endHand()` 新增显式的 `foldWin: contenders.length === 1` 字段（不复用 `handName` 文案做判断逻辑，避免文案和控制流耦合），一路透传到 `game:showdown` socket 事件。客户端按这个字段分流：`foldWin===true` 渲染新的轻量组件 `FoldWinBanner`（居中小卡片，贴近屏幕上方，不挡桌面），否则渲染改造后的 `SettlementModal`（真摊牌专用，强调"赢的牌型"）。
2. **真摊牌延迟出现**：`RoomPage.jsx` 新增 `SHOWDOWN_REVEAL_DELAY_MS=1400`，真摊牌时 `setSettlement` 延迟 1.4 秒执行，把这段时间纯粹留给桌面上的揭示牌，弃牌结束没有牌可看、不需要延迟，`setSettlement` 照常立即执行。需要在 `game:state`/`game:ended` 里 `clearTimeout` 这个挂起的定时器，否则上一手的结算有可能在下一手已经开始后才姗姗来迟地弹出，说的是过期的信息。
3. **结算弹窗从底部抽屉改成顶部锚定**：光缩小高度（62%→42%）并不够——实测发现英雄手牌固定在画布底部，只要弹窗还是从底部升起、且高度够放得下结算内容，迟早会盖到；改成跟 `.fold-win-banner` 同样的定位哲学（顶部锚定，`top:64px`），英雄的手牌无论多少人在场都不会再被挡住。代价：为了不让结算内容随玩家数量无限变高，`max-height` 收紧到 38%（超出部分靠 `overflow-y:auto` 内部滚动）——这样一来人数较多的桌（比如本轮 fixture 用的 6 人局）结算弹窗仍会盖住第 0/1 排对手座位的头像；但用户明确抱怨的是"挡住自己的手牌"，这个已经彻底解决，且延迟出现的 1.4 秒已经给了所有人（含对手）一次完整无遮挡的观牌窗口——这个残留的权衡已知且可接受，未继续深挖。
4. **强调赢的牌型**：`.modal-hand`（"两对，A和K"这类描述）从纯灰色小字改成金色描边的强调 pill，视觉上从"次要说明"变成"赢的理由"这个核心信息。
5. **清理顺手发现的死代码/过期 fixture**：`SettlementModal.jsx` 的 `amtText()` 里 `delta==null→'弃牌'` 分支确认永远不会被触发（`settle[].net` 服务端恒为数字），删掉；`StatesGallery.jsx` 里"结算弹窗"(fixture 7) 传给 `SettlementModal` 的 props 还是旧版 `winner`/`results` 单数签名，跟组件现在的 `winners`/`settle` 数组签名对不上，导致这个预览页早就静默渲染不出东西——一并修正 fixture 数据和调用签名，顺带新增一个"弃牌结束横幅"(fixture 8) 覆盖新组件。

**踩坑记录（跟本轮"气泡遮挡"是同一类根因，第二次踩到）**：`.fold-win-banner` 用 `left:50%; transform:translateX(-50%)` 做居中，但共用的 `slideUp` 入场动画关键帧硬编码了 `transform:translateY(...)`（`fill-mode:both`），一样会把居中位移顶掉——实测第一版横幅整个偏出屏幕右侧。修复方式与 `actionBubbleIn` 相同：把 `slideUp` 的位移也改成用独立的 `translate` CSS 属性，不再跟 `transform` 抢同一个属性。`.action-bar`/`.settlement-sheet` 这两个同样用 `slideUp` 的地方本身没有自己的定位 `transform`，不受影响。

**验证**：Playwright 复现两条真实路径——两人局一方弃牌（验证 `FoldWinBanner` 立即出现、居中、不挡英雄手牌/头像，且 `.settlement-sheet` 计数为 0）；两人局过牌/跟注打到河牌真摊牌（验证揭示牌先出现、结算弹窗延迟约 1.3 秒后才出现、且不挡英雄手牌、`.modal-hand` 正确显示牌型描述）。`?states=` 修好的结算 fixture + 新增的弃牌横幅 fixture 都跑了自动化重叠扫描；服务端 103/103 测试全绿。**踩坑记录（环境）**：编辑 `server/GameEngine.js`/`server/index.js` 后，本地跑着的 `node server/index.js` 进程不会自动 reload（不像客户端有 `vite build` 重新产出静态文件）——第一轮验证 `foldWin` 字段死活收不到，查了半天才想起要重启服务端进程。

### 结算弹窗返工：撤回顶部锚定/独立组件方案，改回底部弹窗+精简信息+英雄手牌按需上挖（用户反馈，2026-07-22）

### 大厅按钮/图标偏小 + 筹码归零处理重新设计：暂停在牌桌等待决策 + 账本永久保留 + 发牌动画放慢（用户反馈，2026-07-21/22）

**背景**：用户一次性提了四类反馈：(1) 大厅页面按钮文字偏小（"再借一底"等）；(2) 顶部菜单按钮的"三条杠"图标相对框体偏小；(3) 筹码清零后系统会把所有人都弹回大厅，希望改成停留在当前牌桌，弹窗提示所有人；(4) 发牌/翻牌动画希望稍微放慢，更有临场感。第(3)点涉及真实架构问题，逐步讨论后确认了完整方案。

**(1)(2) 视觉修复**：`.pr-badge`（"+借一底"/"移出"这类可点击操作）从跟"房主"这类纯信息标签共用的 8px/2px padding 迷你样式，拆出一个新的 `.pr-badge--action` 修饰类（12px 字号、6px 12px padding），可点击的操作跟纯展示信息在视觉上区分开；`.pr-chips` 字号 9px→11px；`.menu-btn` 里的"≡"图标字号 20px→26px。

**(3) 筹码清零处理——完整重新设计**：

*根因排查*（读代码确认，非猜测）：`Room.nextRound()` 只要"有筹码且在线"的玩家数 < 2，就把整个房间打回 `waiting`（大厅）状态，不管当事人有没有做出选择；而 `BustDecisionModal` 只在 `room.status === 'playing'` 时才渲染——所以正好撞上这个"整桌打回大厅"的场景时，弹窗压根不会出现，大家是被无声地扔回大厅的。3 人以上桌若清零后还剩 ≥2 人有筹码，现有逻辑本身没问题（清零者能看到弹窗、其他人牌局不受影响），只是其他人完全没有任何提示。

*方案讨论过程*：先提出按场景区分（能继续打的桌子只给清零者弹窗+其他人轻提示，卡住的桌子才全员弹窗），用户明确要求"不管牌桌能不能继续，其他玩家都应该看到弹窗"——于是统一成一条规则：**一手结束后，只要有人筹码清零，就先暂停发下一手，等这个人决策完再继续，不区分人数场景**。弃牌文案"对手全部弃牌"同一轮也被指出有视角歧义（"对手"是相对于读的人而言，赢家、输家、旁观者看到同一句话理解不一样），改成"其他人全部弃牌"（相对界面上紧邻的、显式命名的赢家而言，任何视角看含义都唯一确定）。

*决策*：
1. **服务端新增 `Room.awaitingBustResolution` 标志**：`index.js` 新增 `tryAdvanceIfClear(room)`，是"是否真的可以发下一手"这个决策的唯一出口——检查 `room.players` 里有没有 `chips===0 && !left` 的人，有就设 `awaitingBustResolution=true` 并广播暂停状态（不调用 `nextRound()`），没有才继续走 `nextRound()`。`advanceRoom()`（结算全员确认后的推进点）、`player:rebuy`、新增的 `player:leave-room`/`room:leave-for` 都收敛到这一个函数。
2. **离开不再删除玩家行**：`Room.removePlayer(id)` 改名 `markLeft(id)`，只打 `left:true` 标记，不再从 `this.players` 里过滤掉——`nextRound()`/`startGame()` 的人数/发牌逻辑都加上 `!left` 排除条件，`getLobbyState()` 把 `left` 字段带给客户端。`RoomManager.leave()`（被 `room:kick`、大厅断线超时、新的主动离开事件共用）内部改调 `markLeft`。房间"是否已废弃可删除"的判断从 `players.length===0` 改成 `players.every(p=>p.left)`。`restart()` 是唯一真正清空历史的地方——额外把 `debt`/`left` 也一起重置，明确"重开一局"和"有人中途离开"是两件不同的事。
3. **两个新 socket 事件**：`player:leave-room`（自己主动离开，账本仍保留这行）——弃牌弹窗的"退出对局"、等待弹窗自己的"退出"按钮、大厅原有的"退出房间"按钮统一收敛到这一个事件，不再单纯依赖断线宽限期。`room:leave-for`（仅房主可用，帮一个清零后不响应的玩家做"退出对局"的决定，且校验目标必须正是当前卡住暂停的那个人）——UI 上复用"断线时行动方卡住"那套已有的顶部提示条+可点击文字链接模式（"帮TA弃牌"→"帮TA离开"），保持交互一致，用户特意点名要求这样做。5 分钟安全超时也从只覆盖断线行动方扩展到覆盖清零未决策，超时后系统自动按"离开"处理。
4. **客户端**：`BustDecisionModal` 去掉第三个"旁观留下"选项（用户明确要求只保留两个）；新增 `BustWaitModal`，给除清零者以外的所有人看，文案"等待 [姓名] 决策中……"，带一个"退出"按钮（这是查看者自己要不要退出，不是投票让清零者继续与否）；两个弹窗都是 `.modal-overlay` 样式，桌面本身继续在底下可见，不会跳转到大厅视图（因为 `room.status` 全程保持 `'playing'`，`inGame` 判断不受影响）。`Lobby.jsx` 的座位列表/空位数/开局人数门槛都改成基于"过滤掉 `left` 之后"的玩家数组；账本（`LedgerModal`）保持展示全部玩家（含已离开的），加一个"已离开"小标签。

*踩坑记录（两处真实 bug，都是实测才发现的）*：
- **第一处**：最初把"是否有人清零"的判断直接放在 `room.players` 上检查，但筹码同步（把 `game.players` 的最终筹码写回 `room.players`）只发生在 `nextRound()` 内部——而判断"要不要调用 `nextRound()`"这件事本身就得先知道有没有人清零，形成先有鸡还是先有蛋的问题。结果是：第一版代码里这个暂停机制自己检查的是**同步之前的旧筹码**，永远不会触发，一个直接复现"两人局清零后立刻 game:ended"的服务端测试因为改动了断言而侥幸"通过"了（实际是暂停从未生效，而不是测试验证了正确行为）。改法：把筹码同步单独拆成 `Room.syncChipsFromGame()` 方法，在 `advanceRoom()` 里、判断暂停之前先显式调用一次。
- **第二处**：修完上面那处后，"清零后选择再借一底"又卡在了"弹窗不消失、牌局不推进"——因为 `tryAdvanceIfClear()` 被再借一底的 handler 第二次调用时，又顺带把 `nextRound()` 自己内部那份（历史遗留、面向"直接单测调用 `nextRound()`"这种用法设计的）筹码同步逻辑跑了一遍，用的是那把**已经打完、没被触碰过**的旧 `GameEngine` 实例，把刚借的筹码覆盖回了 0。改法：`rebuy()` 在给 `room.players` 加筹码的同时，也顺手把同一份数字写回 `this.game.players` 里对应的那个（如果还存在的话），让两份数据保持一致，不管后续谁再重新同步一次都不会互相打架。
5. **发牌/翻牌动画放慢**：`GameTable.jsx` 的 `DEAL_STEP`（每张牌之间的间隔）0.07s→0.09s；`.card-deal` 单张牌落地动画 0.4s→0.48s（`DEAL_CARD_DURATION` 常量同步改成 0.48，顺带修正了这个常量原本就跟 CSS 实际值不一致的旧漂移——注释写的是 0.35 但 CSS 一直是 0.4）；`.flip-reveal`（公共牌翻牌）0.52s→0.62s。

**验证**：服务端新增/重写共 6 条测试（暂停触发、再借一底解除暂停并直接发下一手、主动离开解除暂停+账本保留、房主代为处理+权限校验、非房主调用被拒绝），109/109 全绿。真机验证用直接起一个独立测试服务器实例（复用 `createServer()`，而不是打真实的 dev server）+ Playwright 双页面，直接操作服务端 `room` 对象制造清零场景（比反复真实全下更稳定可靠）：清零者弹窗（含正确文案）+ 等待者弹窗（含正确文案"等待 房主 决策中……"）同时验证通过；三条解除路径（再借一底/自己离开/房主代离开）都验证了状态正确清零、双方全程保持在牌桌视图（`.table-canvas` 可见，未跳转大厅）；离开后账本里该玩家行仍存在且带"已离开"标签；大厅座位列表验证已离开玩家不再出现在座位里。

### 真机三连报：结算显示两家矛盾结果 + 弃牌后仍能看到摊牌 + 每手结束疑似回大厅（用户反馈，2026-07-22）

**背景**：用户带着一张真机截图反馈三个问题，明确要求"从系统全盘的角度看这几个场景，不要打补丁式的"。逐一排查后，两个是读代码就能确认根因并修复的真实 bug，第三个排查后没能在正常/清零两种场景下复现，需要用户提供更多信息才能继续定位。

**Bug A（已修复）：边池里"筹码不够格"跟"全员弃牌"被同一个标签盖住**。截图里同一手结算显示两个"赢家"，一个标着真实牌型"Three of a Kind, A's"，另一个标着"其他人全部弃牌"——这在同一手牌里逻辑上不可能同时成立，用户由此怀疑"结算逻辑有 bug、该一家赢却算成两家分"。读 `GameEngine._endHand()`/`_buildSidePots()`/`_determineWinners()` 后确认：筹码分层（边池）本身的分钱逻辑是对的（这确实是两个不同玩家合法赢下两个不同池层，属于正常的不等额全下场景），**但 `_determineWinners()` 判断"这层只有一个人有资格"时，不管这个人是因为"全场只剩他没弃牌"还是"这层边池筹码额度只有他够格（别人没弃牌，只是筹码不够跟到这一层）"，一律贴"其他人全部弃牌"的标签**——后者其实是一次真实摊牌，只是恰好在某个边池层没有对手陪跑，理应显示真实牌型。修法：把"是否整手都是弃牌获胜"这个判断从"每一层"提到"整手牌"层面（`_endHand` 里算一次 `isFoldWin`，传给每一层的 `_determineWinners`），只有真正整手弃牌收场时才用弃牌文案，其余情况即使某层只有一人够格也照常算出真实牌型描述。

**Bug B（已修复）：弃牌之后仍能看到对手后续摊牌**。`GameEngine.getStateForPlayer(playerId)` 原来的可见性判断是"只要 `phase==='showdown'` 就把所有人的底牌发给这个 viewer"，完全没考虑这个 viewer 本手是不是已经弃牌——用户反馈"弃牌就是弃牌了，怎么还能看到别人的牌"，这条反馈本身合理，直接采纳：viewer 本手若已弃牌，即使别的玩家真的打到了摊牌，也不再把别人的底牌发给这个 viewer（自己的牌不受影响，永远可见）。

**Bug C（排查中，未能复现，需要用户补充信息）：用户反馈"这次更新之后每一轮结束都要回大厅、房主要重新点开始游戏"**。系统性排查过程：
1. 直接读 `tryAdvanceIfClear`/`nextRound` 代码，没找到"正常手牌（无人筹码归零）也会触发回大厅"的路径——`awaitingBustResolution` 只在有人 `chips===0 && !left` 时才为 true。
2. 用 Playwright 连打 3 手完全正常的对局（双方一路过牌/跟注到摊牌，没有全下、没有人筹码归零），全程 `room.status` 保持 `'playing'`、`awaitingBustResolution` 保持 `false`、客户端从未跳转大厅——**证明正常手牌不会触发这个问题**，跟 37/39 号任务里已经验证过的清零场景（弹窗、暂停、三条解除路径全部正确）合在一起看，没有找到能让"每一轮"都回大厅的代码路径。
3. 因此怀疑用户实际遇到的是以下之一，但需要用户确认才能继续：(a) 真的每手都有人筹码归零（清零决策弹窗其实出现过，但被当成"这就是回大厅"的一部分没细看），(b) 手机切后台/锁屏导致 WebSocket 断线，`nextRound()` 里"断线玩家不算 active"这条本来就存在、跟这次改动无关的既有逻辑，在两人对局时只要有一方断线就会让"有筹码的在线玩家数<2"从而回到大厅（这条逻辑不是本轮新增的，但本轮之前从未在两人对局里被这么频繁触发过，值得怀疑）。**这一条明确记录为待用户补充信息后继续排查，不在没有复现的情况下瞎猜着"修"。**

**验证**：Bug A 新增服务端回归测试——构造一个真实的 3 人不等额全下场景（A 全下 1000、B 短码全下 300、C 弃牌），让 A 在没人对该层弃牌、纯粹因为筹码额度独占的边池层里赢下真实牌型而非弃牌标签；Bug B 新增两条测试，分别验证"本手已弃牌的 viewer 在摊牌阶段看不到任何人的底牌"和"没弃牌的 viewer 摊牌阶段仍能正常看到"。服务端 112/112 全绿（含一个跟本次改动无关、已确认是随机发牌偶发平局导致的既有 flaky 测试，重跑三次稳定通过）。

## 结算等待期断线不应自动推进（Bug3 结构性修复，2026-07-23）

**背景**：用户持续反馈一个 bug（前两个 session 未能复现），第三条线索"只有单挑时才会出现，房主每次弃牌都要退出"。逐一排查代码所有路径后定位到根因。

**根因**：`server/index.js` 的 `disconnect` handler 在 settlement wait（筹码结算等待全员确认）期间存在一段特殊的处理逻辑——`dropFromSettlementWait()` 立即把断线玩家从"需要确认"名单里移除，如果因此导致剩余玩家都已确认，则立即调用 `advanceRoom()` 推进到下一手。这是整个断线处理体系里**唯一不符合"断线只标记 connected:false，不推动游戏状态"这个设计原则的地方**。

手机上的完整触发链：
1. 房主弃牌 → 单挑手牌结束 → `beginSettlementWait()`（双方都要点"我知道了"）
2. 对手先点了"我知道了"
3. 房主的手机端 WebSocket 短暂断开（iOS Safari 切后台/网络切换——手机上极其常见）
4. 服务端 `disconnect` handler → `dropFromSettlementWait(房主Id)` → 所有人"确认"了 → `advanceRoom`
5. `advanceRoom` → `nextRound()` → `active = players.filter(p => chips > 0 && connected !== false)` → 房主 `connected = false` → 只有 1 人活跃 → **游戏结束！**
6. 房主 socket 重连后收到 `game:ended` → 回到了大厅

**决策（结构性修复，不是打补丁）**：

把 settlement wait 阶段的断线处理拉回统一的"断线只标记 connected:false，不推动游戏状态"规则：

1. **删掉** `disconnect` handler 里 settlement wait 的 `dropFromSettlementWait` + `advanceRoom` 特殊处理。断线仅广播最新 connected 状态给全房间，不推进游戏。
2. **重连恢复**：`room:sync` 检测到房间处于 settlement wait 时，向重连的 socket 重新下发 `game:showdown` 事件，让结算弹窗重新出现。
3. **存储 showdown 数据**：`Room` 新增 `lastShowdown` 字段存储最近一次 showdown 结果，以便重连时恢复。
4. **踢人可解锁**：`room:kick` 踢人后若 settlement wait 被解除了，自动推进。

**为何这是结构性修复，不是打补丁**：

| 方案 | 做法 | 问题 |
|---|---|---|
| Patch 式 | 在 settlement wait 里判断 heads-up 特殊处理 | 加 if，其他人数下同样问题不改 |
| 结构性（本决策） | 删掉 settlement wait 这条特殊分支，和其他 mid-game 断线统一对待 | 一个规则管全场，一致性 |

这个修复的意义是：当前代码已经有了一套完善的"mid-game 断线 ≠ 离开"的规则（第九轮之前修的），settlement wait 阶段的断线是这套规则唯一的例外。去掉这个例外，把整个游戏生命周期的断线统一成一个规则，不需要任何特殊判断。

**受影响代码**：
- `server/index.js`：`disconnect` handler、`room:sync` handler、`handleActionResult`、`advanceRoom`
- `RoomManager.js`：新增 `lastShowdown` 字段

**验证方法**：用集成测试验证 settlement wait 期间断线不会导致 advanceRoom（当前测试 `integration.test.js` 里有相关的断言，需更新预期行为）。

### 房主可主动结束对局 + 自动展示账本（用户反馈，2026-07-24）

**背景**：用户指出缺少一条"结束对局"的流程——现在只有"筹码不足自动暂停回大厅"（弹个 toast）和"重新开始"（清空筹码重开一晚），没有"房主主动决定今晚就打到这了，把账结清"这条路径。账本（`LedgerModal`）虽然一直存在，但要主动点菜单才能看到，没人会在真正散场那一刻想起来点。

**决策（已用 AskUserQuestion 跟用户确认）**：
1. 触发方式：房主在菜单（≡）里新增"结束游戏"一项，弹确认框（跟现有"退出游戏"同一套 UI 模式），点确认后结束——**不**在筹码不足时自动触发，那条路径维持现状不变。
2. 结束后的状态：**自动弹出账本**（不需要再手动点菜单），房间本身回到大厅（`status:'waiting'`），筹码/账本数字保留，房主之后仍可以点"重新开始"开新的一晚——不是销毁房间，也不是重置数字。

**实现**：
- 服务端新增 `room:end-game`（host-only，校验同 `room:restart`）：`syncChipsFromGame()` 把正在进行的这手筹码落盘 → `game=null, status='waiting'` → 广播 `room:state` + 一条带 `hostEnded:true` 标记的 `game:ended`，reason 用"房主结束了本局对局"，跟筹码不足那条 reason 区分开
- 客户端 `game:ended` 处理器：`hostEnded` 为真时额外 `setShowLedger(true)`，其余清理逻辑（清 gameState/settlement 等）复用现有分支，不新增专门的"结束态"组件
- `GameTable` 菜单新增 host-only 的"结束游戏"行 + 确认弹窗，复用 `.modal-btns`/`.modal-btn-danger` 现成样式

**为什么不做成新组件/新页面**：结束后就是"回到大厅 + 账本自动弹出"，这两个状态（大厅、账本）本来就存在且行为已验证过，不需要再造一个"对局结束页"。

## Risks / Trade-offs

- **内存存储** → 服务重启丢失所有房间。缓解：提示用户游戏中途不要刷新页面；MVP 阶段可接受。
- **单进程 Node.js** → 并发房间多时可能成为瓶颈。缓解：小规模使用，无需水平扩展。
- **无断线重连** → 玩家断网后无法恢复。缓解：自动将断线玩家设为 fold，游戏继续。
- **手牌强度判断复杂** → 摊牌时需要比较多种牌型。缓解：使用成熟的牌型比较算法（如 `pokersolver` 库）。
