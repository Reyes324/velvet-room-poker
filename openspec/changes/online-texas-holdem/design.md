## Context

全新项目，从零构建一个支持多人在线实时对战的德州扑克 Web 应用。目标是轻量、无需注册，通过分享链接即可开始游戏。玩家数量 2-9 人，需要低延迟的实时状态同步。

## Goals / Non-Goals

**Goals:**
- 浏览器直接访问，无需安装任何客户端
- 房主创建房间并分享链接，朋友点击即可加入
- 完整德州扑克规则（No-Limit Hold'em）
- 实时同步所有玩家操作和牌局状态
- 移动端友好的响应式 UI

**Non-Goals:**
- 用户账号/登录/持久化排行榜
- 真实货币/积分系统
- AI 对手（Bot）
- 观战模式
- 游戏进行中的断线重连（MVP 阶段，断线自动 fold，下局重连）——注意：这条只覆盖"游戏进行中"；大厅/等待阶段的断线有独立的宽限期处理，见下方「大厅断线宽限期」决策，两者不是一回事
- ~~游戏中途加入新玩家（游戏进行中拒绝加入，等下局）~~ —— 已被下方「游戏中途加入 + 筹码归零个人决策」决策取代：用户实测反馈希望房主开局后朋友仍能通过链接加入，改为允许中途加入、从下一手自动入座
- ~~观战模式~~ —— 中途加入等待入座、筹码归零后选择"旁观留下"，两者都需要"能看牌局但不参与"的旁观视图，因此这条 Non-Goal 被下方决策顺带实现，不再是有意排除的范围；仍然不做的是"纯观战、从不下场"的独立入口（没有人会以旁观者身份创建/加入房间——旁观状态只能从"中途加入等待"或"筹码归零"这两个具体场景派生）
- 行动超时倒计时对其他玩家可见（延后决策）

## Decisions

### 前后端架构：单体 Node.js + React

**选择**: Express + Socket.io 后端，React 前端，单仓库部署。

**理由**: Socket.io 提供开箱即用的 WebSocket + 降级支持，适合小团队快速开发。单体部署简化运维，无需分离前后端服务。React 便于管理复杂的实时 UI 状态。

**备选方案**: Next.js（SSR 复杂度不必要）、纯 WebSocket（需自行处理重连/心跳）。

---

### 游戏状态管理：服务端权威

**选择**: 所有游戏逻辑和状态只在服务端计算，客户端只渲染和发送操作指令。

**理由**: 防止作弊（客户端不能看到其他玩家手牌）、避免状态不一致。服务端是唯一真相来源。

**备选方案**: 客户端计算 + 服务端验证（逻辑重复，风险更高）。

---

### 房间标识：6 位纯数字码

**选择**: 6 位随机数字（如 `482931`）作为房间 ID。

**理由**: 纯数字比字母数字更容易口头分享（"四八二九三一"），朋友局场景下常见。原方案是字母数字混合，经用户反馈后改为纯数字。

---

### 数据结构：游戏状态对象

服务端维护每个房间一个 `GameState` 对象：

```
GameState {
  roomId, phase (waiting/preflop/flop/turn/river/showdown),
  players: [{ id, name, chips, holeCards, bet, status, isDealer }],
  communityCards[], pot, currentBet, activePlayerIndex,
  deck (服务端私有，不广播)
}
```

广播给玩家时，隐藏其他玩家手牌（只发送 `{ hidden: true }`），只对摊牌阶段或弃牌以外的玩家展示。

## User Scenarios（已确认）

### S1：正常开局
1. 房主打开链接 → 输入昵称 → 创建房间 → 复制邀请链接发朋友

### S3：筹码归零 —— 已被下方「筹码归零 Rebuy（借一底）」章节取代并实现，本节仅保留历史背景
- 玩家输光筹码 → 当局结束后自动移出下一局
- 触发 `game:ended` 事件 → 客户端显示结束消息
- ~~未来可扩展：重新买入（rebuy）按钮，暂不实现~~ **已实现**，见「筹码归零 Rebuy（借一底）」章节
- **Bug 修复记录**：`nextRound()` 因筹码不足结束时，服务端曾经只广播 `game:ended`、没有再广播 `room:state`，导致大厅里落败玩家看不到自己最新的 0 筹码和"+借一底"按钮（要手动刷新才会更新）。已修复为 nextRound 后统一广播 room:state，见 tasks.md 10.2

### S4：游戏中途加人
- 新玩家尝试加入进行中的房间 → 返回错误"游戏已开始，无法加入"

### S6：行动超时 ~~（已取消）~~
- **决策：整个超时机制已移除** — 用户明确不需要，朋友局场景下强制限时体验差

### 游戏桌面布局：椭圆形桌面（行业标准）

**选择**：用绝对定位实现椭圆桌面，英雄（本人）固定在底部中央，对手围绕上半弧排列。

**理由**：调研 PokerStars/GGPoker/PartyPoker 及多个开源项目（Tehes/poker 等）后确认，椭圆桌面是移动端扑克的行业标准。原 CSS Grid 布局缺乏空间感，不符合用户心理模型。

**关键设计决定**：
- ~~对手牌背（face-down back）保留显示~~ **已被 v3 取代**：rail 布局评审中用户要求移除对手座位的暗牌占位框，对手座位只显示 头像 + 角标 + 筹码，手牌仅在摊牌时揭示（见下方"牌桌布局精修"）。
- 英雄座位不显示小牌背，其手牌改为底部大尺寸展示（消除重复）
- 下注筹码从座位内移出，浮动在座位与桌心之间（模拟筹码推向中央）

### 牌桌布局精修：头像坐上椭圆 rail（v3 预览评审）

**选择**：对手头像圆心落在椭圆边界上（跨坐 rail），所有对手座位统一渲染，5 人沿上半弧均匀分布，英雄固定底部。

**理由**：v2 预览里头像被放在手机黑边（座位完全在椭圆外），"牌桌"与"玩家"视觉割裂。调研真实德州 App（PokerStars/GG）确认头像应贴在桌沿。统一渲染消除"顶部座位特殊、侧座裸头像"的不一致。

**配套决定**：
- 对手座位信息精简为：头像（汉字身份首字）+ 位置角标（D/SB/BB，贴头像右下角）+ 单行筹码码片（仅金额，无姓名）。对手筹码量是必看战术信息，故保留码片；姓名由头像字符承载，去掉以减标签。
- 街道指示（翻牌前/翻牌/转牌/河牌）以低调小字放在底池区上方，不放回顶栏。顶栏只留菜单 + 余额。
- 摊牌揭示牌放在头像朝外一侧，不侵入椭圆中心。
- **行动倒计时仍不加**（再次确认 S6 决策）：活跃玩家只用金色脉冲高亮，无任何计时 UI。

### 视觉信息层级与配色（v3 预览评审）

**问题**：早期每个元素（位置角标、下注、后手、底池、街道）都是"带边框的小药丸"，且颜色多（角标白/黄/蓝），无主次，视觉嘈杂。

**决定（建立单一层级，去边框）**：
- **底池**：去掉边框/背景，纯发光金色大数字浮在桌面，作为唯一视觉焦点。
- **街道指示、后手**：去边框，纯文字。
- **下注筹码**：唯一保留"筹码感"的元素——一个金色圆形筹码 token + 数字，无外框。
- **位置角标 D/SB/BB**：统一为单一象牙色"按钮"样式（取消白/黄/蓝三色），小尺寸贴头像边缘。
- **金 vs 冷色分工**：金色专用于"在玩的钱"（底池、下注）；**后手（stack）改用冷色 platinum `#9FB0AC`**，作为辅助信息退后一层，与金色拉开差距。
- **金额字体两级层级**：
  - **主角数字（底池、结算赢得金额）用展示衬线体 `DM Serif Display`** —— 敦实优雅，作为全场焦点数字有高级牌局质感。
  - **功能性小额（后手、下注、余额、结算明细）保持等宽体 `Space Mono`** —— 对齐工整、读作"账本/钱"。
  - 这是有意的两级层级（焦点数字优雅 / 功能数字工整），非"随便混字体"。演进过程：Cinzel（数字花哨被否）→ 一度全 Space Mono（底池放大偏技术感）→ 现选 DM Serif Display 给主角数字。
  - Cinzel 仅保留给装饰性标题与房间码（非金额）。

### 扑克牌牌面设计（v3 预览评审）

**选择**：牌面升级为接近真实扑克牌的高级质感。

**关键细节**：
- **对角双角标**：左上 + 右下（旋转 180°）各一组"点数 + 小花色"，符合真牌惯例（原仅左上单角标显得简陋）。
- **无边框**：曾加内描线框，用户不喜欢框，已去掉——牌面只留暖白纸面 + 角标 + 中心花色，干净（即样式 A）。
- **暖白纸面**：象牙白渐变 + 内高光，配合更讲究的投影。
- **花色配色**：红 `#B3271E`（更沉稳）、黑 `#1F2A33`（略冷的近黑）。
- **点数字体用 `Oswald`（窄体）**：解决 Georgia 下两位数"10"偏扁的问题，condensed 让数字紧凑利落；花色符号仍用原字体渲染。
- 牌背：深绿渐变 + 金色 ✦ 暗纹。
- **弃牌玩家**：不再整体半透明（会透出桌面，观感差），改为**实心、去饱和、压暗**的头像 + 下方"弃牌"文字说明。视觉清晰，用户明确认可。
- 翻牌前不显示公共牌占位实底（牌未发出），改为极淡的虚线空位，减少"框框"噪声。

### 样式架构：设计系统为唯一源，预览即状态画廊（方案 B）

**问题**：早期 `states-preview.html` 预览**脱离 design token 体系**单独手调，调出的最终值困在预览内联 CSS 里，没沉淀回 tokens/组件；生产组件又用另一套 class（`card--lg` 等）"翻译"预览，每次翻译走样，预览失去"一次确认直接落地"的意义。

**选择（方案 B）**：
- **唯一事实源 = `tokens.css`（值） + 共享组件 CSS**（`client/src/styles/velvet.css`，用预览那套 class，引用 token）。
- **预览 = 状态画廊**：`states-preview.html` 改为 link 共享 CSS、用同样 class 摆出所有状态，不再内联另写一份。
- **生产 = 同一套 CSS + 数据**：Card/PlayerSeat/RoomPage 直接 import 共享 CSS、输出预览的 class 与 DOM 结构。
- 三者同源，改一处 CSS/token，画廊与生产同步，永不翻译。

**落地步骤（一次性返工）**：① 把已确认预览的最终值回收进 tokens + `velvet.css`；② 预览改造为画廊；③ 生产组件改用预览 class + import velvet.css。

**理由**：用户的诉求是"轻量预览确认设计 → 直接复制进生产"。只有让生产**消费**预览的 CSS（而非重写），该工作流才成立。详见协作手册「设计迭代要在设计系统上做」。

**备选（未选）**：方案 A —— 直接把预览 `<style>` 当生产 CSS，token 退役。更快但丢失 token 抽象（语义色名/主题化），用户看重 token 故选 B。

### 筹码归零 Rebuy（借一底）

**决策（Q1 已确定）**：采用**选项 B** —— 支持 rebuy，玩家输光后留在房间，在大厅等待阶段自助借一底。

**规则**：
- 一底 = ¥1,000（与初始买入相同）
- 每次 rebuy += 1,000 chips，debt += 1,000（无上限，由玩家自行决定）
- 时机：~~仅在 waiting 状态（大厅）下允许 rebuy，游戏进行中不可借~~ —— 已被下方「游戏中途加入 + 筹码归零个人决策」决策取代：门禁改成"本人 chips===0 即可借，跟房间整体状态无关"，因为 3 人以上桌子里游戏进行中归零的玩家在旧规则下要等到整桌结束才能借，体验缺口太大
- 仅输光筹码（chips=0）的玩家才会看到 rebuy 按钮（本人可见）
- debt > 0 时，所有人都能在大厅 + 游戏桌看到该玩家的"借了¥X"债务状态

**数据模型**：在 `players` 对象添加 `debt: 0` 字段，通过 `getLobbyState()` 广播给所有玩家。

**nextRound 行为变更**：不再自动移除 chips=0 玩家，改为仅把有筹码的玩家传入 `GameEngine`；chips=0 玩家在大厅旁观等待 rebuy，rebuy 后下一局参战。

**UI**：
- 大厅：chips=0 时本人看到"+借一底"按钮；全员看到"借¥X底"橙色 badge
- 游戏桌：PlayerSeat 暂不展示 debt（大厅展示已足够；避免游戏桌信息过载）

### 移动端设计规范：Scale-to-Fit（统一适配方案）—— ~~已被下方「顶部/底部贴边 + 仅中间牌桌缩放」取代~~，本节仅保留历史背景

**选择**：以 375×812（iPhone X）为标准设计画布，通过 JS 计算 scale 系数（`Math.min(vw/375, vh/812)`）自适应所有设备，`vh` 取当前实际可见视口高度（`visualViewport.height`，回退 `innerHeight`）。

**规则**：
- 游戏画布固定 375×812，所有绝对坐标相对此画布
- `vh/812`（完整高度，不裁切）：scale 必须让整个 375×812 画布完全落在当前可见视口内
- `vh` 必须用「当前实际可见」的高度，不能用 `window.screen.height`（物理屏幕高度）——地址栏/浏览器 UI 出现时 `screen.height` 仍是满屏值，比真正可见区域（`innerHeight`/`visualViewport.height`）大，会算出偏大的 scale，把画布顶部推出可见视口之外（`.stage-wrap` 是底部锚定，超出部分从顶部溢出）。单挑（1v1）时唯一的对手座位正好在椭圆桌面正上方顶点，是第一个消失在屏幕外的元素
- `transform-origin: bottom center` + `.stage-wrap { align-items: flex-end }`：游戏从底部锚定，确保操作栏（action bar）在任何设备上始终可见
- 同时监听 `resize`、`orientationchange`、`visualViewport.resize`，地址栏出现/隐藏时重新计算 scale
- 适配结果：iPhone SE（375×667）有约 17px 侧边空白（可接受），现代机型（390×844+）填满屏幕；地址栏可见时画布整体等比缩小而非被裁切

**背景**：早期版本存在两个根本性布局问题：
1. `RoomPage.css` 中的旧 `.lobby { min-height:100dvh }` 与 velvet.css 的绝对定位大厅打架，导致"开始游戏"按钮被推出屏幕
2. 移动端没有统一的适配规范，不同屏幕表现不一致

**移除 RoomPage.css**：发现 `RoomPage.css` 是旧设计系统遗留的冲突文件（`.lobby-code`、`.start-btn`、`.table-view` 等已废弃 class），全部移除，Toast 样式迁入 velvet.css，统一单源架构。

**踩坑记录**：中途曾改用 `window.screen.height` 替代 `innerHeight`（意图是让地址栏出现/隐藏时 scale 不跳动、避免小屏出现黑边），并同时把裁切余量从 `vh/712` 收紧为 `vh/812`。这两个改动叠加后，只要浏览器地址栏可见（`screen.height > innerHeight` 的场景，移动端打开页面的常态），scale 就会算大，画布顶部（含单挑唯一对手座位）整段推出可见视口——即用户反馈的"单挑时对手内容飘出手机屏幕"。修复：`vh` 改回读实际可见视口（`visualViewport.height` 优先，`innerHeight` 兜底），让画布等比缩小以完整容纳，而不是用偏大的物理尺寸硬算再被裁切。

### 移动端布局重构：顶部/底部贴边 + 仅中间牌桌区域缩放（用户实测反馈第八轮，2026-07-19）

**背景**：上面的 Scale-to-Fit 方案有个没意识到的副作用——用户反馈手机上牌桌两侧有明显留白，宽度没拉满。原因：`.game-stage` 是整个 375×812 画布作为一个整体用单一 `scale = Math.min(vw/375, vh/812)` 缩放。如果浏览器（尤其是某些 App 内置浏览器）上下 UI 常驻占用空间较多，可视区域的宽高比会比 375:812 更"矮胖"，`vh/812` 会小于 `vw/375`，缩放取小值时高度是瓶颈，宽度被迫跟着一起缩小——两侧因此留白，即使宽度本来有富余空间没用上。

**选择**：不再对整个画布做单一等比缩放，改成"顶部状态栏、底部操作栏各自按自己的实际内容渲染、贴边显示，宽度永远等于视口宽度；只有中间的牌桌区域（椭圆桌+座位+英雄手牌）跟随剩余空间弹性缩放"。本质是从"整体 transform:scale"改成"顶固定高度 + 底固定高度 + 中间 flex:1 弹性区域"的标准 flex 布局。

**实现**：
1. `.game-stage` 从固定 375×812 尺寸 + `transform:scale()` 改成 `width:100%; height:100dvh; display:flex; flex-direction:column;`
2. `.top-bar` 从绝对定位改成真实 flex 子元素（`flex:0 0 auto`），`padding-top` 用 `max(env(safe-area-inset-top,0px),14px)` 处理刘海屏安全区，永远贴顶、永远满宽
3. 新增 `.table-zone`（`flex:1 1 auto; min-height:0; overflow:hidden;`）+ `.table-canvas`（固定 375×`TABLE_REF_H`(610) 的设计画布，通过内联 `transform:scale()` 缩放并居中）——牌桌椭圆、座位、英雄手牌全部挪进这个新画布，`GameTable.jsx` 新增 `useTableScale(ref, refW, refH)` hook，用 `ResizeObserver` 直接测量 `.table-zone` 的实际渲染尺寸算缩放系数，不再依赖 `window.innerWidth/innerHeight` 或 `visualViewport`——这一层的旧坑（`screen.height` vs `innerHeight`）从根上不存在了，因为缩放对象从"整个视口"变成了"一个真实测量到的 DOM 容器"
4. `.action-bar`/`.waiting-bar` 同理改成 flex 子元素（`flex:0 0 auto`），永远贴底、永远满宽，`padding-bottom` 叠加 `env(safe-area-inset-bottom)`
5. `.lobby`（大厅玩家列表）同样从绝对定位改成 `flex:1 1 auto`——本来就是纯文字列表，不含需要保持比例的几何绘制，不需要参与缩放，直接享受原生响应式布局
6. `seatPositions()` 的椭圆参数（`cy`）、`table-oval` 顶部偏移、`hero-section` 的 `bottom` 值、`CARDS_SIDE_BELOW_Y` 阈值，都因为参考画布从 812 缩到 610（顶部状态栏不再占用画布内空间）做了等比重新推导

**意外的架构收益**：`.action-bar` 现在是牌桌画布**外面**的独立 flex 兄弟节点，展开加注面板导致操作栏变高时，会自动挤压 `.table-zone` 的可用高度、让整个牌桌画布等比缩小，物理上不可能再出现"加注面板盖住英雄手牌信息条"——第二/三轮反馈里为了避免这个重叠专门做的像素级精调（`hero-section: bottom:178px→148px` 来回调整、压缩加注面板高度）现在不再需要手动兜底，是这次重构顺带解决的。

**验证**：Playwright 4 种视口（含完全复现用户截图问题的"浏览器 UI 占用高度"430×777 场景，以及更极端的 430×650）下确认顶部/底部宽度精确等于视口宽度、零元素溢出可见区域；加注面板展开截图确认全宽显示且不再与手牌重叠。

### 顶部对手头像仍会飘出 + 牌桌区域改成横纵独立缩放（用户实测反馈第九轮，2026-07-19）

**背景**：上面的重构上线后用户反馈"中间顶部的头像又有点飘出去"，同时提出一个更根本的问题：牌桌区域自己（`.table-canvas`）内部仍然用的是单一 `Math.min(w/refW, h/refH)` 等比缩放——跟这一整轮问题的根因（顶部/底部贴边重构之前，整页缩放同样的算法）完全一样，只是把"两侧留白"这个症状从整个页面收缩到了牌桌这一个区域内部。当浏览器 UI 占用高度较多、高度是瓶颈时，牌桌区域两侧一样会重新出现留白。

**头像飘出根因（用 Playwright 量出来的真实数字，不是猜的）**：`seatPositions()` 的椭圆 `cy=215, ry=195`，最顶上那个对手座位中心点落在 `cy-ry=20`，跟 `table-oval` 的 CSS 顶部偏移 `top:20px` 精确相等——而这个 20px 同时也几乎正好等于头像本身的半径（默认头像 40px 直径，半径 20px）。也就是说，最顶上那个头像的设计半径和到画布顶边的留白**从一开始就是零边距**，任何一点亚像素取整、边框宽度、位置徽章，都会把它推到 `.table-zone` 的 `overflow:hidden` 裁切线以上。实测在高度是瓶颈的视口下（如 430×777、430×650、375×667）该座位真的会被裁掉 ~1px——量级虽小，但证明这不是设备特例，是画布尺寸设计上就没留够余量，在越极端的视口下只会越明显。

**选择**：
1. 牌桌区域缩放从"单一等比 `scale(s)`"改成"横向、纵向各自独立缩放 `scale(scaleX, scaleY)`"——`scaleX` 永远等于容器实际宽度 ÷ 参考宽度（牌桌横向永远贴满，不会再有留白），`scaleY` 独立按容器实际高度 ÷ 参考高度计算，不再取两者较小值。代价：在跟设计稿参考比例（375:610）差异较大的极端视口上，头像会有肉眼可辨但不夸张的轻微椭圆化（实测最极端场景约 20% 形变），换来牌桌横向永远填满——用户已在这一轮反馈中明确认可这个取舍（"宽的时候可以扁一点，窄的时候可以瘦一点"）。
2. `seatPositions()` 的 `ry` 从 195 收紧到 180（`table-oval` CSS 的 `top`/`height` 同步从 20/390 改成 35/360），把最顶上那个座位的中心点从 y=20 推到 y=35，给头像半径腾出约 15px 的真实缓冲，不再是设计上的零边距。`rx`（横向半径）、`cy`（中心点）不变，牌桌整体形状只是纵向略微更圆一点，肉眼几乎看不出差异。

**验证**：Playwright 4 种视口 × 双方视角实测头像包围盒坐标（不是目测），修复前在 3/4 视口下最顶上的对手头像确有 ~1px 裁切，修复后全部归零；同时验证牌桌椭圆左右两侧留白始终等于设计稿里 18px 按 `scaleX` 换算后的固定内边距（跟视口宽度成正比、左右对称），而不再是缩放算法带来的、跟高度瓶颈相关的不对称留白。

### 撤销「牌桌内容随非等比缩放变形」的取舍 + 菜单按钮放大（第十二轮，2026-07-19）

**背景**：上一条决策里用户认可的取舍——浏览器往外拉伸、`scaleX`≠`scaleY` 时头像轻微椭圆化——实测下来并不是"轻微"：`.table-canvas` 上的 `scale(scaleX, scaleY)` 作用于整个子树，不只是头像，牌、底池数字、街段标签、筹码文字全都跟着两轴不同比例被拉伸/压扁，浏览器越往宽拉变形越明显。用户明确表示这样不对：牌桌圈（位置/形状）可以拉宽，但牌、数字、标签这些具体内容不能变形。等于推翻上一轮"用户已认可此取舍"的决定。

**选择**：把"位置铺开"和"内容渲染"拆成两层，不再共用同一个非等比变换：
1. 位置层不变——`.table-canvas` 依旧用 `scale(scaleX, scaleY)`，座位/牌桌圈的位置和形状继续非等比铺满容器（牌桌横向永远贴满这个既有优点保留）。
2. 内容层新增统一缩放 `scaleUniform = Math.min(scaleX, scaleY)`，并算出反向抵消系数 `csx = scaleUniform/scaleX`、`csy = scaleUniform/scaleY`（写成 CSS 变量 `--csx`/`--csy`）。凡是渲染实际内容的节点（牌桌内容包裹层 `.table-oval-content`、每个座位 `.player-slot`、`.hero-section`）在继承父级非等比缩放的基础上，追加 `scale(csx, csy)`——两次缩放相乘后视觉上等价于单一的 `scale(scaleUniform, scaleUniform)`，牌、头像、文字因此始终保持原比例，同时位置仍然跟随非等比的 `scaleX/scaleY` 铺开。
3. 代价：`scaleUniform` 取两轴较小值，极端宽高比视口下内容会比"贴着某一轴拉满"更小一些（周围留白变多），但换来的是牌桌上任何文字/图形都不再变形——这次明确是用户要的取舍方向，跟上一轮相反。

**顺带修复**：`.menu-btn`（左上角菜单按钮）从 32×32px 放大到 44×44px（触屏最小可点击尺寸的行业标准），字号从 15px 提到 20px。

### 游戏中途加入 + 筹码归零个人决策 + 账本视图（用户实测反馈第十轮，2026-07-19）

**背景**：用户提出两个交互问题。(1) 房主开局后，朋友能否仍通过邀请链接加入，从下一手自动入座？现状是硬拒绝（`Non-Goals` 原先明确排除这条）。(2) 某玩家筹码归零后，现在的行为是"等到全桌只剩 <2 个有筹码的人时，整个房间掉回大厅，游戏对所有人中断"；用户希望改成"只有归零本人看到一个决策弹窗（借一底 / 旁观留下 / 离开），其他人的牌局不受影响地继续（前提是还有 ≥2 人有筹码——单挑归零依然只能整体暂停，这是德州扑克规则本身决定的，不是要修的 bug）"。用户同时要一个账本视图（初始筹码/已借入/当前筹码），随时可查，不限于大厅。

**排查发现（两轮 Explore + 一轮 Plan 子代理逐行核实源码）**：`GameEngine.js` 的 `players` 数组是每手从 `Room.players` 重新过滤构建的一次性快照，从未支持、也不需要支持"插入正在进行的一手"——两个功能的插入点天然都在"两手之间"，因此 `GameEngine.js` 和 `server/index.js` 完全不需要改动，改动全部集中在 `RoomManager.js` 的两处门禁 + 客户端渲染路径。过程中发现一个已经能在生产环境触发的潜伏 bug：`GameTable.jsx` 的 `const me = ordered[0]`（未检查"我是否真的在这一手玩家列表里"）在归零场景下会把某个真实对手的座位错误地渲染成"我"（名字标错、位置徽章全错），而不是报错或掉回大厅——这两个新功能会让这个潜伏 bug 从"几乎撞不到的边界情况"变成"每次归零/中途加入都触发"，本轮一并修复。

**选择**：
1. `Room.addPlayer`（`RoomManager.js`）去掉"仅 `status==='waiting'` 才能加入"的拒绝，只保留 9 人满座 + 重复 id 检查；新玩家照常 `chips:1000`，`nextRound()` 本来就按 `chips>0` 从 `this.players` 全量重建下一手的参战名单，中途加入的人自动从下一手生效，不需要新增"待入座"标记字段。
2. `Room.rebuy`（`RoomManager.js`）门禁从"仅 `status==='waiting'`"改成"仅本人 `chips===0`"，与房间整体状态无关——见上方「筹码归零 Rebuy」决策的更新。
3. 顺带修复庄位轮转的潜伏 bug：`dealerIndex` 是对每次重新过滤出的 `active` 数组取模，数组大小/顺序因中途加入或归零排除而变化时庄位会不连续地跳。改成记 `dealerId`（上一手庄家的玩家 id），下一手在新数组里找这个 id 的位置、庄位顺延到其后的下一个座位；找不到（该玩家已离场）则退回座位 0。这个改动不属于用户原始请求，是本轮排查中顺带发现并修复的，因为这两个新功能会让牌桌人数变动的频率大幅提高，不修的话下一次实机测试大概率会把它当成新 bug 报回来。
4. `Room.getLobbyState()` 新增 `startingChips` 字段，账本视图直接读取，不在客户端重复维护一份 1000 常量。
5. 客户端：`RoomPage.jsx` 新增 `amPlaying`（我是否在当前这一手 `gameState.players` 里）判断，替代过去"有 `gameState` 就当作我在游戏里"的隐式假设；`GameTable.jsx` 新增旁观渲染路径（不再有效座位时不得回退到 `ordered[0]`），座位几何新增一个"没有英雄锚点、把所有人均匀分布在完整椭圆上"的变体；筹码归零的检测用 `prevChipsRef` 追踪本人筹码 `>0→0` 跳变触发决策弹窗，跟本次会话之前给发牌动画加的 `justDealt`/`justRevealed` 判定是同一套手法。选择"旁观留下"后，footer 保留一个常驻的"+借一底"入口（不会被卡死出不来）。
6. 顶部 ≡ 菜单从"点击直接弹退出确认框"改成两行小弹层（账本 / 退出游戏），`GameTable.jsx` 和 `Lobby.jsx` 两处入口保持一致。

**Non-Goals 联动**：见上方 Non-Goals 小节，"游戏中途加入新玩家"和"观战模式"两条已相应划掉。

**验证**：服务端单元测试（75/75）覆盖 addPlayer 中途加入、rebuy 的 chips===0 门禁正反两个方向、dealerId 轮转在"庄家中途归零离场"场景下的正确性。

**踩坑记录：这个沙盒环境的多 page 硬限制**——实现时想按最初计划写 3 个真实玩家身份的 Playwright 场景（已开局 2 人 + 中途加入的第 3 人），结果发现这个沙盒环境存在一个跟本次改动完全无关的限制：单个测试进程里第 3 个浏览器 page（不管是否属于同一个 context）几乎总是卡死在 `page.goto` 上——用 3 个完全不含任何游戏逻辑的空白页复现出同样的结果，确认是环境本身的资源上限（很可能是代理层的并发连接数限制），不是应用代码问题；这也解释了旧版 S4 测试（3 个 context）为什么本来就是本 session 记录在案的已知偶发失败之一。应对方式：① 服务端集成验证改用 socket.io 服务端自带的 `/socket.io/socket.io.js` 客户端脚本，在已有的 page 里开一条裸 socket 连接模拟"第三个玩家"，只用 2 个真实 page 就验证了 `room:join` 在游戏进行中真的会被服务端接受、返回 1000 筹码、不在当前这一手里；② 客户端旁观渲染路径（不会把座位错标成"我"、footer 三态、两个新弹窗）改用项目已有的 `?states=N` 开发自检画廊（`fixtures.js`/`StatesGallery.jsx`），单 page 直接给 `GameTable`/新组件喂真实 props，覆盖力度不打折扣，还避开了这个环境限制；③"归零玩家单独借入、其他人不受影响"这条由 `RoomManager.test.js` 的专门单测覆盖（比端到端 UI 测试更精确，也完全不受这个环境限制影响）。

### 行动指示气泡统一 + 发牌质感/动画补全（第十三轮，2026-07-19）

**背景**：用户反馈三个问题。(1) "轮到谁"的雷达波纹圈视觉上像"一个断续、线框不完整的圈"，不想要，要求统一改成气泡样式，且气泡不应该像现在这样会消失。(2) 牌背太薄，感觉不出"派了牌"。(3) 公共牌/对手手牌的发牌动画不够有"发牌"的动作感，而且对手手牌发完之后就彻底看不见了。

**排查发现**：(1) 读代码发现实际上存在两套独立机制——对手下注一直有一个不会消失的持久 `.bet-chip` 气泡，但 hero 自己下注只有一条 1.6 秒后淡出的文字提示，从来没有对应的持久气泡；"不同人眼中不统一"说的正是这个不对称，不是幻觉。(2) `PlayerSeat.jsx` 里对手的两张背面小牌只在 `dealing`（发牌那一瞬间，约 1 秒）渲染，动画播完 `dealing` 变 false 后这段 JSX 直接不渲染了，直到摊牌才重新出现——"对手好像没有牌"是真实的展示 bug，不是错觉。

**选择**：
1. 去掉 `.seat.is-active::before/::after`"雷达波纹圈"整套（含 `@keyframes activeRing`），头像本身的描边发光（`.is-active .avatar` 的 border+box-shadow 脉冲）保留——那是另一套机制，不是用户描述的"圈"。
2. `GameTable.jsx` 给 hero 补上跟对手完全同源的 `.bet-chip`（同一套"朝向池心方向平移"的计算），从此 hero 自己下注也有一个不会消失的持久气泡，跟对手视角完全一致。文字类的"说话"气泡（跟注/加注/弃牌/过牌/ALL IN）保持原有的淡出行为不变——用户明确说的是"气泡表示放筹码"这一种要改成不消失，不是所有反馈都要常驻。
3. `.c-back` 从纯色渐变+单个星标，加上内嵌高光/暗部阴影模拟出厚度（不是纯平矩形），再叠一层低透明度（4-5%）菱格纹理——是材质模拟，不是页面装饰条纹，且只作用于 28-84px 的卡片元素本身。
4. `cardDeal` 动画从"原地缩放淡入"改成"从右上方带旋转位移滑入再摆平"，更接近发牌手部动作的观感；社区牌本来就有的逐张交错延迟（`communityDealDelayFor`）沿用不变，这次只是让每一张落地时的动作更明显。
5. 对手的两张背面小牌改成整手牌局期间常驻显示（`hasCards = gamePhase !== 'waiting'`），只在刚发牌那一下播放飞入动画，之后静止显示，弃牌后隐藏，摊牌时切换成揭示样式（原逻辑不变）。
6. 连带问题：手牌常驻之后，会跟同样默认贴在头像正上方的"说话"文字气泡打架。新增 `bubbleStyle()`：手牌在上方时文字气泡堆叠到手牌上方；手牌因为贴近顶部边缘被推到侧边时（`cardsSide` 命中），文字气泡改用头像正上方的默认位置——两者互补，不会同时占用同一个位置。
7. 密集桌（7-9 人）连带问题：Playwright 实测 9 人桌场景后发现相邻座位的常驻手牌会互相重叠甚至被裁到视口外——`.game-stage--dense` 新增头像 38→34px、手牌 28×40→21×30px、`.reveal` 间距收紧的密集桌专属尺寸，重叠消除。

**验证**：`?states=` 自检画廊逐态截图核对（含临时插入又移除的"密集桌测试"9 人场景、临时插入又还原的 hero 下注场景），控制台无报错；受限于自检画廊只渲染单帧静态 state（`actionBubbles` 靠运行时状态变化才会产生，静态 fixture 天然产生不了），气泡与常驻手牌"不重叠"这条主要靠代码走查 + 上面第 6 点的互补定位逻辑保证，未能在自检画廊里截到两者同框的真实帧，后续接入真实房间实测时应留意确认。

### 修复发牌动画从未真正播放过的 bug，并纠正上一轮的错误结论（第十七轮，2026-07-19）

**背景**：用户反馈"公共牌没有动画"。这轮之前（第十六轮，见 tasks.md 24.3）曾用 Web 搜索核实过德州扑克真实发牌规则（从小盲开始、顺时针、一次一张、两轮、公共牌最后发），对照代码后得出"发牌顺序本来就完全符合规则，不用改"的结论——**这个结论是不完整的**：核实的是发牌*顺序算法*对不对，没有核实动画*到底有没有真的播出来*，这是两件不同的事，后者才是用户真正在说的问题。

**根因（真实的竞态条件，靠证据定位，不是猜的）**：`GameTable.jsx` 里原本有个 `justDealt` 变量，纯粹从一个 ref（`prevPhaseRef`）派生，本质是"单次渲染脉冲"——只在 `prevPhaseRef` 还没追上 `gameState.phase` 的那一次渲染里等于 `true`。问题是：负责把 `prevPhaseRef.current` 更新到最新值的 effect，和负责触发 `setHeroRevealed(false)`（从而让 `dealing` 状态变 true）的 effect，在同一次 mount 的 effect 刷新周期里，按组件里声明的先后顺序**同步依次执行**——前一个 effect 先把 `prevPhaseRef` 追平，后一个 effect 才触发的那次重渲染里，`justDealt` 早就已经变回 `false` 了。也就是说，浏览器从来没有机会真正画出 `justDealt=true` 那一帧——用它控制的公共牌 `card-deal`、hero 自己的初始扣牌 `card-deal`、对手座位飞入 `deal-in` 三处动画，全部被无声跳过，外观上直接是"东西已经在那了"，不报错也不卡住，肉眼很难第一时间发现。

对照组也证实了这一点：对手手牌的 reveal 动画一直是好的，因为它用的是另一个信号——`dealing`（`=!heroRevealed`），一个真正持续存在、跨越整个发牌窗口的 state，而不是像 `justDealt` 那样的一次性脉冲。

**验证方法**：光看代码不够可靠（人眼容易被"逻辑看起来没问题"骗过去），改用真实证据——临时把 `DEAL_STEP`（每张牌之间的间隔）从 0.07 秒调到 4 秒，把整个发牌过程拉长到接近 1 分钟，刷新页面后立刻用 Playwright 读 DOM 的 `getComputedStyle(el).animationName`：公共牌、hero 扣牌、对手座位的 class 里根本没有 `card-deal`/`deal-in`，`animationName` 全是 `none`——实锤。

**选择**：整个删掉 `justDealt` 和它专属的 `prevPhaseRef`，三处动画统一改用已经验证可靠的 `dealing` 状态（新增具名变量 `const dealing = !heroRevealed`，与 `PlayerSeat` 的 `dealing` prop 同名同源）。这是"把还在用旧机制的角落，统一到已经验证 work 的机制上"，不是引入新概念，改动面小、风险低。

**教训**：这次的误判提醒了一件事——核实"规则/算法对不对"和核实"效果有没有真的呈现出来"是两个独立的检查，都要做，不能因为前者通过了就假设后者也没问题。以后遇到"动画/效果好像没生效"这类反馈，第一步应该是用类似的方式（拉长时间窗口 + 读真实 DOM 状态）直接验证效果是否触发，而不是只重新读一遍逻辑代码。

### 牌背改经典蓝格纹 + 账本盈亏合计 + 中文标签 + 撤销结算自动跳转（第十四轮，2026-07-19）

**背景**：用户连续反馈四个点。(1) 牌背用了跟桌布同色系的深绿+金色，用户反馈"不像市面上的扑克牌"，明确要蓝色格纹款。(2) 账本只有"当前"筹码，用户问"当前是什么意思，是不是应该有个最终盈亏"。(3) SB/BB/D 英文缩写希望换成中文"小盲/大盲/庄家"。(4) 亲测发现摊牌结算弹窗即使自己什么都没点，15 秒后也会自动开下一局——跟"必须每个人都点'我知道了'才能继续"的要求矛盾。

**排查发现**：结算的"15 秒自动跳转"不是本轮引入的新代码，是 `server/index.js` 里一段更早就存在的兜底定时器（`settlementFallbacks`），设计初衷是防止有人掉线或卡住导致房间永远卡在结算页——但它是无条件触发的，不管有没有人点确认都会在 15 秒后强制推进，直接绕过了"全员确认"这个产品要求，两者互相矛盾。同时发现结算弹窗上"等待其他人确认（X/Y）"这个数字其实是客户端本地伪造的（`iAmReady ? 1 : 0`，只反映"我自己点没点"），从来没有从服务端同步过其他人的真实确认进度，撤销自动跳转之后这个数字必须是真的，否则用户没法判断到底还要等谁。

**选择**：
1. `.c-back` 从"深绿+金色，低对比度菱格材质感"改成"深蓝渐变+白色格纹，可见对比度的经典款"，白色描边——直接对标市面常见的蓝色扑克牌背，不再往桌布同色系上靠。
2. `LedgerModal.jsx` 新增"盈亏"列，公式 = 当前 − 初始 − 已借（已借的筹码是借来的，不算真赢的，必须扣掉才是真实净输赢），正负用绿/红两色区分，说明文案同步写清楚公式。
3. `PlayerSeat.jsx` 的位置角标从 D/SB/BB 改成中文"庄家/小盲/大盲"，`.pos-badge` 相应加宽。
4. **删掉 15 秒兜底定时器**，结算弹窗不再有任何自动推进——只有两条路能进入下一手：全员真实点击"我知道了"，或者掉线玩家被移出待确认名单之后剩下的人已经都确认了（原有的"掉线不卡死其他人"这条保留，只是不再额外加一个无条件的时间兜底）。`RoomManager.js` 新增 `getSettlementProgress()`，`server/index.js` 在每次有人确认或掉线时把真实的 `{readyCount,totalCount}` 广播给全房间（新增 `game:settlement-progress` 事件），客户端不再自己伪造这个数字。
5. 顺带处理用户"头像和手牌会不会有点大"的顾虑：默认（非密集）尺寸头像 40→36px、对手常驻手牌 28×40→24×34px，密集桌（7-9人）同步收紧到 31px/19×27px，缓解玩家变多时的拥挤感。

**Trade-off（需要用户知情）**：删掉 15 秒兜底定时器意味着——如果某个玩家的确认请求因为网络问题真的没送达服务器（不是他们不想点，而是请求丢了），房间会一直卡在结算页等不到头，不像之前那样最多卡 15 秒就会自动放行。这是用户本轮明确要求的取舍（"必须每个人都点关闭才可以"），但要向用户说明这个代价，让其判断是否需要另一种更温和的兜底（比如卡住时房主可以手动强制推进），而不是完全没有任何安全网。

**验证**：服务端单元测试 75/75 通过（删掉定时器没有破坏任何既有断言）；`?states=` 自检画廊截图确认牌背/中文角标/账本盈亏列视觉效果符合预期。结算无自动跳转 + 真实确认进度这两条依赖真实多客户端 socket 交互，本地自检画廊无法覆盖，留待用户真机联机验证。

### 每局开局发牌动画（用户实测反馈第四轮，2026-07-18）

**背景**：task 9.9 落地的 `deal-in` 只在 `prevPhaseRef.current === null` 时触发——按注释"fires exactly on first mount"是有意为之，但实际效果是发牌动效只在整个会话的第一手牌播放过一次，从第二手开始就没有了。用户要求"每一局都要有"，且要按真实发牌顺序：按座位顺序发牌（从小盲开始）→ 亮出自己的手牌，正式开始这一局。

**决策**：
1. **修复 `justDealt` 只触发一次的问题**：判断条件从 `prevPhaseRef.current === null`（只在组件首次挂载时成立）改成 `prevPhaseRef.current !== 'preflop'`（任何一次"进入 preflop"的转变都成立——包括每一手新牌开局），配合已有的 `prevPhaseRef` 更新逻辑，改一行判断就让动效在每一局都重新播放
2. **发牌顺序 = 真实规则**：新增 `sbFirstOrder()`，以 `gameState.players`（座位固定顺序，不受 `getOrderedPlayers` 为 UI 重排的影响）为基础，从小盲位置开始轮转排序；每人 2 张牌按"轮次×人数 + 座位序号"算 stagger 延迟，等价于真实发牌员一圈一张、发两圈
3. **对手也有牌可看**：新增每个对手座位在发牌期间显示 2 张背面小卡（复用 `.reveal` 的定位槽位，摊牌揭牌和发牌动效互斥不会撞），按上面的顺序 stagger 弹入，营造"每个人都拿到牌了"的真实感；发牌流程结束后统一淡出/消失
4. **英雄手牌先背面落地、再翻开**：英雄的两张牌也先以背面按同样的发牌顺序 stagger 弹入，等全桌的牌都"发完"（`totalDealTime` = 按最后一张牌的延迟 + 单张落地动效时长算出来，不是拍的常量）之后，再统一翻面（复用已有的 `flip-reveal` 3D 翻面动效），翻牌那一刻就是"正式开始这一局"的信号
5. **踩过的坑**：第一版把"翻面倒计时"的 `useEffect` 依赖写成了 `[justDealt]`——但 `justDealt` 本质是"只在某一次 render 为 true 的脉冲"，下一次 render 就变回 false，依赖数组一变化就会立刻触发 effect 的 cleanup，把刚设的 `setTimeout` 取消掉，导致牌永远翻不回正面。改成依赖 `[gameState.phase]`（在整个 preflop 下注轮内这个值是稳定的字符串，不会跟着每次 render 抖动）才对：只在"刚进入 preflop"这一次真正触发，不会被同一 phase 内的后续 render 提前取消

### 发牌动画补全：公共牌也要先扣着发下来，到点再翻开（用户实测反馈第十一轮，2026-07-19）

**背景**：上面第四轮做的发牌动画只覆盖了玩家手牌（英雄+对手），公共牌区域在翻牌前一直是空的虚线占位框，直到翻牌圈/转牌圈/河牌圈实际到来时才凭空"变出"卡牌（配 `flip-reveal` 动效）。用户反馈：应该在给玩家发完手牌之后，公共牌也要用同一套发牌动效摆上桌——只是扣着（背面），等真正轮到那条街要开牌时，再把对应的牌翻面。这更符合真实牌局的观感：庄家一次性把这一手要用到的牌都摆好，该翻的时候才翻，而不是每条街都临时"变"出牌来。

**决策**：
1. **发牌顺序追加一段**：原来"手牌发完 → 英雄翻面"的两段式，改成"手牌发完 → 5 张公共牌扣着摆上（新增）→ 英雄翻面"三段式。公共牌沿用手牌那套 `DEAL_STEP` 间隔的 stagger 节奏，紧接在最后一张手牌落地之后逐张摆出，`totalDealTime`（决定英雄何时翻面）相应顺延，把公共牌摆牌这段时间也算进去。
2. **摆牌之后怎么"扣着"**：公共牌槽位从"没数据就画虚线空框（`.c-empty`）"改成"没数据但已经发过牌 → 画一张背面卡牌（复用 `Card` 组件的 `faceDown`）"；只有在这一手还没开始发牌（`waiting`/刚进房间还没 `justDealt` 过）时才保留虚线空框——否则每一局最初进入时会先闪一下背面牌，不自然。
3. **翻牌时刻沿用现成的 `flip-reveal`**：某条街真正揭晓（`gameState.communityCards[i]` 从没有变成有值）时，槽位从"背面卡牌"切换成"正面卡牌 + `flip-reveal`"——`flip-reveal` 本身的关键帧是从 `rotateY(-90deg)` 转到 `0deg`（见 task 9.9 落地时写的 `flipIn`），本来就是"侧面立起转正面"的视觉，只是过去应用在"从虚线空框凭空出现"上不够真实；现在改成应用在"从一张已经躺在原地的背面卡牌"上，才是名副其实的翻牌动效，这条 CSS 关键帧完全不用改，问题只在于"翻牌之前那张牌在不在"。

**验证**：Playwright 截图逐帧检查"手牌发完瞬间"确认 5 个公共牌槽位已经是背面卡牌（不再是虚线框）；截图检查翻牌圈/转牌圈/河牌圈揭晓瞬间确认对应槽位从背面变正面、其余槽位仍是背面；确认发牌总时长（英雄翻面时机）相应顺延、没有跟对手发牌动效撞在一起。

### 大厅断线宽限期：修复"房间不存在"的根因（用户实测反馈第七轮，2026-07-19）

**背景**：用户反馈"房主创建房间、把链接发给朋友，朋友打开时经常提示'房间不存在'，要重试几次才行"。排查后定位到根因，且顺带修了两个相关的连带 bug。

**根因**：`server/index.js` 的 `disconnect` 处理器过去是这样的：任何 socket 断开，不管房间处于什么状态，立刻 `rooms.leave(myPlayerId)`——如果这个玩家是房间里唯一的人（房主刚建完房、朋友还没加入进来，正好是这个状态），房间会被直接删除（`RoomManager.leave()`：`players.length===0` 就 `this.rooms.delete(code)`）。而"房主创建房间后，切到微信/短信把链接发出去"这个动作，在移动端浏览器上很容易导致标签页被系统挂起、WebSocket 连接短暂断开——即便 socket.io 客户端几秒后会自动重连，服务端在断开的那一刻就已经把房间删没了。朋友点链接进来时，房间已经不存在，房主自己可能都没察觉（页面还停在大厅旧状态，重连后也没有任何机制告诉服务端"我回来了、别删我"）。房主往往通过"重新建一个房间再发一次"来解决，这正好对应用户描述的"重试几次才行"。

**决策**：
1. **大厅阶段断线加宽限期**：只有 `room.status==='waiting'`（大厅/未开局）时才延后处理断线——不立即 `rooms.leave()`，而是设一个定时器，到期时检查这个玩家是否已经用新 socket 重新关联（`socketId` 是否还指向这个已经断开的旧连接），没有才真正移除。游戏进行中的断线依然立即处理（自动 fold + 移除），不受影响，MVP 阶段"游戏中断线不支持重连"的既有决策不变。宽限期时长最初定为 30 秒，用户反馈"可能不够"后改为 **120 秒**（`GRACE_PERIOD_MS`），给切 App 分享链接/临时被打断留更宽松的容错
2. **重连超时的显式报错**：之前如果玩家超过宽限期才回来（房间已经被真正移除），`room:sync` 只是静默返回，客户端会卡在一个数据早已过期的大厅界面上，自己毫无察觉。新增 `room:gone` socket 事件：`room:sync` 找不到房间时显式告知客户端，`RoomPage.jsx` 收到后弹出"重新连接超时，房间已失效，请重新创建或加入"的 toast，2.5 秒后自动跳回首页（跟已有的 `room:kicked` 处理方式一致）
2. **修复连带 bug——`room:sync` 从未真正"重新连接"过**：原来的 `room:sync` 只是读一下房间状态发回去，从来没有调用过 `room.updateSocket()`（更新玩家的 socketId）、`socket.join(room.code)`（重新加入 socket.io 房间广播列表）、也没有设置这个连接自己的 `myPlayerId`。也就是说就算断线重连的宽限期加上了，重连后的客户端也收不到后续广播、它自己下次断线也不会被正确处理——这是一个更早就存在但没被注意到的 bug（正常整页刷新时，旧连接已经在刷新前断开触发了立即删除，恰好掩盖了这个问题；只有"标签页不刷新、底层 socket 断了又连上"这种情况才会暴露）。修复：`room:sync` 现在会完整地重新执行这几步关联
3. **客户端配合**：`RoomPage.jsx` 的 `room:sync` 调用从"只在组件挂载时发一次"改成"每次 socket 触发 `connect` 事件都发一次"（挂载时的首次连接和之后任何一次重连都会触发 `connect` 事件），否则移动端标签页被挂起又恢复时，客户端不会主动告诉服务端"我回来了"
4. **验证方式**：Playwright 直接操作页面里暴露的 socket 实例（`window.__vrSocket`，新增的调试钩子）强制断开/重连，而不是用浏览器级别的离线模拟——`context.setOffline()`/CDP `Network.emulateNetworkConditions` 在这个沙盒环境里会影响到同一个浏览器进程里其他不相关页面的网络，不适合模拟"只有某一个玩家断线"的场景。真实复现了完整场景：房主断线 3 秒（房间未删除）→ 朋友在房主离线期间成功加入 → 房主重连后正确看到朋友已经在大厅里

## Risks / Trade-offs

- **内存存储** → 服务重启丢失所有房间。缓解：提示用户游戏中途不要刷新页面；MVP 阶段可接受。
- **单进程 Node.js** → 并发房间多时可能成为瓶颈。缓解：小规模使用，无需水平扩展。
- **无断线重连** → 玩家断网后无法恢复。缓解：自动将断线玩家设为 fold，游戏继续。
- **手牌强度判断复杂** → 摊牌时需要比较多种牌型。缓解：使用成熟的牌型比较算法（如 `pokersolver` 库）。
